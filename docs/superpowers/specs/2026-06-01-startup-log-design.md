# 启动完成日志整理设计

日期：2026-06-01

## 背景

当前 Python 后端启动日志只输出基础启动信息，缺少类似 Alon 项目的启动完成摘要。需要在 Web 服务、任务调度器、消息监听器启动完成后，按功能阶段输出启动耗时、已加载模块和访问地址，便于开发和排查启动性能。

## 目标

为 `server/python` 的三个启动入口增加结构化启动完成日志，格式参考 Alon 项目：

1. **runserver** - Web 服务
2. **runtask** - 任务调度器
3. **runlistener** - 消息监听器

阶段划分采用按功能分组：

1. 配置加载
2. 基础组件初始化
3. 模块加载与路由注册（Web）/ 模块加载（Task/Listener）
4. 数据初始化（Web）/ 任务调度器启动（Task）/ 监听器启动（Listener）

## 非目标

- 不调整数据库、模块加载、Seed 初始化的现有业务逻辑。
- 不引入第三方依赖。
- 不新增多文档路由，仅输出本项目现有 `/docs` 地址。

## 设计方案

### 新增启动计时工具

新增 `server/python/src/framework/utils/startup_timer.py`，包含：

- `PhaseInfo`：记录阶段名称、耗时和可选明细。
- `StartupTimer`：记录总启动时间和各阶段耗时。
- `StartupTimer.phase(name)`：上下文管理器，用于包裹阶段逻辑并自动记录耗时。
- `StartupTimer.add_detail(key, value)`：为当前阶段追加日志明细。
- `StartupTimer.print_summary(...)`：按 Alon 风格输出最终启动摘要。

### Web 启动流程集成 (application_web.py)

在 `server/python/src/application_web.py` 中集成 `StartupTimer`：

- 在模块加载入口 `create_app()` 中记录模块加载与路由注册耗时。
- 在 `lifespan()` 中记录：
  - 配置加载阶段：标记配置已就绪。
  - 基础组件初始化阶段：数据库引擎、TenantProvider。
  - 数据初始化阶段：执行各模块 seed。
- 在 `lifespan()` 的启动逻辑完成后、`yield` 之前输出启动完成摘要。

### 任务调度器启动流程集成 (application_task.py)

在 `server/python/src/application_task.py` 中集成 `StartupTimer`：

阶段划分：

1. 配置加载
2. 基础组件初始化（数据库引擎、TenantProvider）
3. 模块加载
4. 任务调度器启动（收集并启动各模块的调度器）

在所有任务调度器启动后输出启动完成摘要。

### 消息监听器启动流程集成 (application_listener.py)

在 `server/python/src/application_listener.py` 中集成 `StartupTimer`：

阶段划分：

1. 配置加载
2. 基础组件初始化（数据库引擎、TenantProvider）
3. 模块加载
4. 监听器启动（收集并启动各模块的监听器）

在所有监听器启动后输出启动完成摘要。

## 错误处理

- 阶段计时上下文使用 `finally` 记录耗时，即使阶段抛出异常也能保留已完成阶段信息。
- Seed 初始化继续沿用当前策略：单个 seed 失败只记录异常，不阻止应用启动。
- 单个任务调度器/监听器启动失败只记录异常，不阻止其他模块的调度器/监听器启动。
- 启动失败时不额外伪造"启动完成"日志。

## 测试与验证

  - 阶段耗时被记录。
  - 阶段明细可追加。
  - 摘要输出包含总耗时、阶段、模块和访问地址。
- 运行相关测试：
  - `uv run pytest tests/framework/ -v`
  - 如启动依赖可用，运行以下命令人工检查日志格式：
    - `uv run python manage.py runserver`
    - `uv run python manage.py runtask --module demo`
    - `uv run python manage.py runlistener --module demo`
