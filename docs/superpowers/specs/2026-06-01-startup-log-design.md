# 启动完成日志整理设计

日期：2026-06-01

## 背景

当前 Python 后端启动日志只输出基础启动信息，缺少类似 Alon 项目的启动完成摘要。需要在 Web 服务启动完成后，按功能阶段输出启动耗时、已加载模块和访问地址，便于开发和排查启动性能。

## 目标

为 `server/python` 的 FastAPI Web 启动流程增加结构化启动完成日志，格式参考 Alon 项目，阶段划分采用按功能分组：

1. 配置加载
2. 基础组件初始化
3. 模块加载与路由注册
4. 数据初始化

## 非目标

- 不调整数据库、模块加载、Seed 初始化的现有业务逻辑。
- 不修改 `runtask`、`runlistener` 的启动日志。
- 不引入第三方依赖。
- 不新增多文档路由，仅输出本项目现有 `/docs` 地址。

## 设计方案

### 新增启动计时工具

新增 `server/python/src/framework/utils/startup_timer.py`，包含：

- `PhaseInfo`：记录阶段名称、耗时和可选明细。
- `StartupTimer`：记录总启动时间和各阶段耗时。
- `StartupTimer.phase(name)`：上下文管理器，用于包裹阶段逻辑并自动记录耗时。
- `StartupTimer.add_detail(key, value)`：为当前阶段追加日志明细。
- `StartupTimer.print_summary(...)`：按 Alon 风格输出最终启动摘要。

### Web 启动流程集成

在 `server/python/src/application_web.py` 中集成 `StartupTimer`：

- 在模块加载入口 `create_app()` 中记录模块加载与路由注册耗时。
- 在 `lifespan()` 中记录：
  - 配置加载阶段：标记配置已就绪。
  - 基础组件初始化阶段：数据库引擎、TenantProvider。
  - 数据初始化阶段：执行各模块 seed。
- 在 `lifespan()` 的启动逻辑完成后、`yield` 之前输出启动完成摘要，确保日志出现在应用可服务前。

### 输出格式

示例：

```text
============================================================
Demo API 启动完成！
总启动耗时: 2.357 秒
启动阶段耗时:
  阶段1 (配置加载): 0.012秒
    - 配置文件: application.yml
  阶段2 (基础组件初始化): 0.234秒
    - 数据库: PostgreSQL
    - TenantProvider: 已注册
  阶段3 (模块加载与路由注册): 0.156秒
  阶段4 (数据初始化): 1.955秒
完成时间: 2026-06-01 17:30:45
🔌 已加载模块: 3 个
   - demo
   - iam
   - tenant

访问地址: http://127.0.0.1:8000
API 文档: http://127.0.0.1:8000/docs
============================================================
```

## 错误处理

- 阶段计时上下文使用 `finally` 记录耗时，即使阶段抛出异常也能保留已完成阶段信息。
- Seed 初始化继续沿用当前策略：单个 seed 失败只记录异常，不阻止应用启动。
- 启动失败时不额外伪造“启动完成”日志。

## 测试与验证

- 新增 `StartupTimer` 单元测试，验证：
  - 阶段耗时被记录。
  - 阶段明细可追加。
  - 摘要输出包含总耗时、阶段、模块和访问地址。
- 运行相关测试：
  - `uv run pytest tests/framework/ -v`
  - 如启动依赖可用，运行 `uv run python manage.py runserver` 人工检查日志格式。

## 范围说明

本设计只覆盖 Python 后端 Web 服务启动日志，不影响 Rust、前端或其他子项目。
