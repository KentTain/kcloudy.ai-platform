## 发现的问题及修复

### 问题 1: 事件监听器测试事件循环不匹配 (已修复)

**文件**: `server/python/tests/ai/e2e/test_event_listeners.py`

**错误信息**:
```
RuntimeError: Task got Future attached to a different loop
```

**根因**: `test_plugin_installation_failed_event` 在 `asyncio.sleep(3)` 后使用 `e2e_session.execute()` 查询数据库，但 session 的 asyncpg 连接与当前事件循环不匹配。

**修复方案**:
1. 验证查询使用新创建的 `AsyncSession(bind=e2e_engine)`，避免旧 session 的连接跨循环
2. `test_installation_record` fixture 使用 `commit()` 替代 `flush()`，确保新 session 可见数据

### 问题 2: 前端 IAM fixtures 导出缺失 (已修复)

**文件**: `web/vue/tests/iam/e2e/fixtures.ts`

**错误信息**:
```
SyntaxError: The requested module '../../iam/e2e/fixtures' does not provide an export named 'userLoginViaAPI'
```

**根因**: `iam/e2e/fixtures.ts` 从 `tenant/e2e/fixtures.ts` 导入 `userLoginViaAPI` 并重命名为 `iamUserLogin` 导出，但 AI 插件测试需要原始名称 `userLoginViaAPI`。

**修复方案**: 在 IAM fixtures 中同时导出 `iamUserLogin` 和 `userLoginViaAPI`。

### 问题 3: 前端测试选择器不匹配 (已修复)

**文件**: `web/vue/tests/ai/e2e/plugin-list.spec.ts` 等

**根因**: 测试使用 `h2` 和通用选择器，但实际页面使用 `data-testid="plugin-manage-page"` 和 shadcn-vue 组件。同时 UI 登录方式在 Docker 环境中不稳定。

**修复方案**:
1. 使用 API 辅助登录 (`userLoginViaAPI`) 替代 UI 登录
2. 更新选择器匹配实际页面结构
3. 增加容错处理（`isVisible().catch(() => false)` 和 `test.skip()`）

### 问题 4: 数据库迁移未执行 (已发现并解决)

**现象**: API 返回 `relation "ai.plugin_installations" does not exist`

**根因**: 运行 E2E 测试前 AI 模块的数据库迁移未在当前数据库中执行。

**解决**: 执行 `uv run python manage.py db migrate --all -y`

## 前端 AI E2E 测试状态

前端测试在 Docker 环境中遇到以下限制：
1. **内存不足**: Chromium 浏览器在容器内运行容易崩溃
2. **IPv4/IPv6 兼容性**: Playwright 的 `request` 上下文默认连接 IPv6，但 Vite 服务只监听 IPv4
3. **登录后数据加载**: 登录成功但插件列表为空（未安装插件）

已修复的问题（代码层面）：
- `plugin-fixtures.ts` 导入错误
- 所有 AI E2E 测试文件的选择器更新
- API 辅助登录替代 UI 登录

## 建议

1. **API Key 管理**: 为 E2E 测试配置专用的测试 API Key，避免依赖外部服务可用性
2. **Docker 资源**: 增加容器内存限制至少 4GB，以支持 Chromium 浏览器运行
3. **IPv6 支持**: 在 Vite 配置中添加 `host: '::'` 支持 IPv6，或在 Playwright 配置中强制使用 IPv4
4. **CI 集成**: 将后端 E2E 测试加入 CI 流水线，前端 E2E 测试需要专用运行环境
