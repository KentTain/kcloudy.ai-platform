# 2.  通用模块（`tenant`）的架构设计需求

## 通用模块（`tenant`）的目录结构

@server\\{技术栈}\src\tenant 目录下是后端不同技术栈的实现方法（java、netcore、python、rust），通用模块结构如下：

```text
server/技术栈/              # 不同技术栈 后端项目根目录，例如：java、netcore、python等
├── src/framework/            # 通用模块
│   ├── core/                  # 核心框架（路径、常量、单例等）、统一接口定义（存储、缓存、队列、发布订阅、锁）
│   ├── common/                # 通用模块（异常、上下文、日志等）
│   ├── config/                # 统一配置模块（基于 YAML 文件的读取）
│   ├── database/              # 通用数据库访问的实现，例如：树结构、Tenant实现
│   ├── storage/               # 存储的实现，例如：本地、MinIO、阿里云等等
│   ├── cache/                 # 缓存的实现，例如：本地、redis等等
│   ├── queue/                 # 队列的实现，例如：mysql/pgsql行级锁、redis等等
│   ├── lock/                  # 锁的实现，例如：mysql/pgsql行级锁、redis等等
│   ├── pubsub/                # 发布订阅的实现，例如：redis、rabbitMQ等等
│   ├── tenant/                # 多租户管理（包括每个租户使用的资源配置，例如：数据库、存储、缓存、队列、发布订阅、模型等）
│   ├── utils/                 # 工具函数，包括：字符串、时间日期、列表、枚举
│   ├── CLAUDE.md                # framwork 开发指南
│   └── README.md                # framwork 说明文档
├── tests/framwork/              # framwork 测试文件
```

## ## 多租户（src/tenant）设计需求

租户模型的设计，租户有相关资源的配置信息（数据库、存储、缓存、队列、发布订阅、模型），可参考：[D:\Project\kcloudy\java\Common\kc.framework\src\main\java\kc\framework\tenant](D:\Project\kcloudy\java\Common\kc.framework\src\main\java\kc\framework\tenant)

整体流程如下：
- 前端用户登录完，会有个默认的租户（也可切换不同的租户），会向后端传递一个-header：`x-tenant-id`，里面记录了：tenant_id；
- 后端根据tenant_id，可以从数据库中获取tenant的信息，保存至线程安全的全局上下文当中；
- 再根据租户的数据库配置，动态加载数据库数据源，实现不同租户的动态数据源切换功能；
- 再次请求，可以从全局上下文中获取已有的tenant的信息；
