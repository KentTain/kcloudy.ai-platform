# 1. 统一基础设施及后端实现

整体采用pgsql+pgvector、redis（cache、queue、pub/sub）、miniIO，使用MVC架构的技术实现方式

# 2.  统一架构设计

@server\ 目录下是后端不同技术栈的实现方法（java、netcore、python、rust），整体的项目结构如下

```text
server/config/              # 配置文件（基于 YAML）
│   ├── application.yml        # 基础配置
│   ├── application-local.yml.examples  # 本地开发配置（示例）
│   └── application-local.yml  # 本地开发配置  
server/技术栈/              # 不同技术栈 后端项目根目录，例如：java、netcore、python、rust
├── src/模块/                 # 不同模块 后端源码，例如：framework、examples、auth、user
│   ├── components/            # 可插拔组件框架
│   ├── controllers/           # API 控制器
│   ├── services/              # 业务逻辑层
│   ├── models/                # 数据库模型
│   ├── schemas/               # Controller使用的DTO模型
│   ├── configs/               # 配置管理（基于 YAML 文件的读取）
│   ├── common/                # 通用模块（异常、上下文等）
│   ├── db/                    # 数据库引擎配置
│   ├── migrations/            # 数据库迁移文件
│   ├── core/                  # 核心框架（路径、环境、单例等）
│   └── utils/                 # 工具函数
├── tests/模块/              # 不同模块 测试文件
├── config/                  # 配置文件（引用 @server/config/ 下的文件）
├── scripts/                 # 开发脚本
├── CLAUDE.md                # 不同技术栈 开发指南
├── README.md                # 不同技术栈 说明文档
```

# 3. 开发及说明分层

@server\ 下的CLAUDE.md、README.md，只需要整体的说明，并引用每个技术栈的CLAUDE.md、README.md，不需要每个技术栈的详细

# 4. 统一健康检查

每个技术栈都要提供-健康检查 API 端点

# 5. 统一配置

 @server/技术栈/config/ 下的配置文件是引用 @server/config/ 的配置文件
