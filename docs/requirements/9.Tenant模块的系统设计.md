# 租户系统设计文档

## 1. 概述

### 1.1 设计目标

重新设计租户系统的数据模型及功能，实现模块化、可扩展的租户管理体系。

### 1.2 核心架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                        模块定义层 (Tenant Module)                     │
│  ┌──────────┐  ┌─────────────┐  ┌──────────────────┐                │
│  │  Module  │──│ ModuleMenu  │  │ ModulePermission │                │
│  └──────────┘  └─────────────┘  └──────────────────┘                │
│       │                                                              │
│       │ ┌──────────────┐  ┌─────────────────────┐                   │
│       └─│  ModuleRole  │──│ ModuleRolePermission│                   │
│         └──────────────┘  └─────────────────────┘                   │
│       │                                                              │
│       │ 分配模块（自动同步）                                          │
│       ▼                                                              │
├──────────────────────────────────────────────────────────────────────┤
│                        租户实例层 (IAM Module)                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐│
│  │   Menu   │  │Permission│  │   Role   │  │ UserRole /           ││
│  │(ref_id)  │  │(ref_id)  │  │(ref_id)  │  │ RolePermission       ││
│  └──────────┘  └──────────┘  └──────────┘  └──────────────────────┘│
└──────────────────────────────────────────────────────────────────────┘
```

### 1.3 模块划分

| 模块 | 职责 |
|------|------|
| **Tenant 模块** | 租户管理、资源配置、模块定义、模块分配 |
| **IAM 模块** | 用户管理、角色管理、权限管理、菜单管理（租户实例层） |

### 1.4 数据层级

1. **模块定义层**（全局）：定义模块包含的菜单、权限、默认角色
2. **租户实例层**：租户实际使用的菜单、权限、角色（通过 `tenant_id` + `ref_id` 关联）

### 1.5 默认角色

每个模块定义两个默认角色：
- **系统管理员**：拥有模块所有权限（读 + 写）
- **普通用户**：只有查看权限（只读）

## 3. 功能设计

### 3.1 设计决策汇总

| 决策项 | 结论 |
|--------|------|
| 资源测试连通性 | 需要，每个类型加 `test-connection` 端点 |
| 资源与租户绑定 | 一对一，当前模型已支持 |
| 模块定义→租户实例同步 | 自动同步 |
| 租户定制菜单/权限 | 完全禁止，纯镜像 |
| 取消模块分配 | 级联删除，有用户使用时禁止取消 |
| 功能架构方案 | 职责分离：Tenant 模块发事件，IAM 模块监听处理 |

---

### 3.2 资源管理功能

资源配置独立存在，多个租户可共享同一资源配置。

#### API 端点

所有资源配置遵循统一的 CRUD 模式，以数据库配置为例：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/tenant/resource-configs/databases` | 数据库配置列表 |
| POST | `/api/v1/tenant/resource-configs/databases` | 创建数据库配置 |
| GET | `/api/v1/tenant/resource-configs/databases/{id}` | 获取单个配置 |
| PUT | `/api/v1/tenant/resource-configs/databases/{id}` | 更新配置 |
| DELETE | `/api/v1/tenant/resource-configs/databases/{id}` | 删除配置 |
| POST | `/api/v1/tenant/resource-configs/databases/{id}/test-connection` | 测试连通性 |

其余四类（storage/cache/queue/pubsub）端点结构相同，替换路径中的 `databases` 即可。

#### 业务规则

1. **删除保护**：如果配置已被租户引用（FK），禁止删除，返回错误提示
2. **密码加密**：创建/更新时，密码字段使用 AES 加密存储，读取时脱敏返回
3. **测试连通性**：仅验证连接参数是否正确，不修改任何数据；超时设为 5 秒

#### 测试连通性实现要点

| 资源类型 | 测试方法 |
|----------|----------|
| Database | 尝试建立连接并执行 `SELECT 1` |
| Storage | 检查 bucket 是否存在且可访问 |
| Cache | 执行 `PING` 命令 |
| Queue | 根据类型：Redis 用 PING，RabbitMQ 检查队列列表 |
| PubSub | 根据类型：Redis 用 PING，Kafka 检查 topic 列表 |

---

### 3.3 分配租户资源

租户与资源配置通过 FK 直接关联，每种资源类型一个租户只能绑定一个配置。

#### API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/tenant/tenants/{id}/resources` | 获取租户的资源绑定情况 |
| PUT | `/api/v1/tenant/tenants/{id}/resources` | 更新租户的资源绑定 |

#### 请求体示例

```json
{
  "db_config_id": "uuid-xxx",
  "storage_config_id": "uuid-yyy",
  "cache_config_id": "uuid-zzz",
  "queue_config_id": null,
  "pubsub_config_id": null
}
```

#### 业务规则

1. **整体更新**：资源绑定采用整体替换策略（PUT），一次请求可同时更新多种资源
2. **解绑传 null**：将某个配置设为 `null` 即解绑
3. **引用校验**：绑定的配置 ID 必须存在，否则返回 404
4. **无级联影响**：解绑资源不影响租户其他数据，仅断开关联

### 3.5 菜单及权限的统一管理

本节描述**模块定义层 → 租户实例层**的自动同步机制。

#### 领域事件设计

Tenant 模块发布以下事件，IAM 模块监听处理：

| 事件名 | 触发时机 | 携带数据 |
|--------|----------|----------|
| `ModuleAssigned` | 租户分配模块 | tenant_id, module_id |
| `ModuleUnassigned` | 租户取消模块分配 | tenant_id, module_id |
| `ModuleMenuCreated` | 模块新增菜单 | module_menu_id |
| `ModuleMenuUpdated` | 模块菜单更新 | module_menu_id |
| `ModuleMenuDeleted` | 模块菜单删除 | module_id, menu_code |
| `ModulePermissionCreated` | 模块新增权限 | module_permission_id |
| `ModulePermissionUpdated` | 模块权限更新 | module_permission_id |
| `ModulePermissionDeleted` | 模块权限删除 | module_id, permission_code |
| `ModuleRoleCreated` | 模块新增角色 | module_role_id |
| `ModuleRoleUpdated` | 模块角色更新 | module_role_id |
| `ModuleRoleDeleted` | 模块角色删除 | module_id, role_code |
| `ModuleRolePermissionChanged` | 模块角色权限变更 | module_role_id |

#### 同步处理器逻辑（IAM 模块监听）

**ModuleAssigned 事件处理：**

```
1. 查询模块的所有菜单、权限、角色
2. 为该租户创建 Menu、Permission、Role 记录（ref_id 关联模块定义）
3. 创建 RolePermission 关联
4. 按 parent_id 重建菜单树结构
```

**ModuleMenuCreated 事件处理：**

```
遍历所有已分配该模块的租户：
  1. 创建 Menu 记录，ref_id = module_menu_id
  2. 根据 module_menu.parent_id 找到对应的租户菜单 parent_id
```

**ModuleMenuDeleted 事件处理：**

```
遍历所有已分配该模块的租户：
  1. 删除 ref_id = module_menu_id 的 Menu 记录
  2. 级联删除子菜单
```

**ModulePermissionDeleted 事件处理：**

```
遍历所有已分配该模块的租户：
  1. 删除 ref_id = module_permission_id 的 Permission 记录
  2. 级联删除 RolePermission 关联
```

**ModuleRoleDeleted 事件处理：**

```
遍历所有已分配该模块的租户：
  1. 检查是否有 UserRole 关联
     - 有用户使用 → 级联删除 UserRole 关联后继续
  2. 删除 ref_id = module_role_id 的 Role 记录
  3. 级联删除 RolePermission 关联
```

#### 租户实例层 API（只读）

租户实例层的菜单/权限/角色数据由同步机制写入，租户管理员只能查询：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/iam/tenants/{tenantId}/menus` | 租户菜单树 |
| GET | `/api/v1/iam/tenants/{tenantId}/permissions` | 租户权限列表 |
| GET | `/api/v1/iam/tenants/{tenantId}/roles` | 租户角色列表 |
| GET | `/api/v1/iam/tenants/{tenantId}/roles/{roleId}` | 角色详情（含权限列表） |

### 3.6 分配租户模块

租户通过模块分配获得模块功能，同时自动同步菜单、权限、角色到租户实例层。

#### API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/tenant/tenants/{id}/modules` | 获取租户已分配的模块列表 |
| POST | `/api/v1/tenant/tenants/{id}/modules` | 为租户分配模块 |
| DELETE | `/api/v1/tenant/tenants/{id}/modules/{moduleId}` | 取消租户的模块分配 |

#### 分配请求体

```json
{
  "module_id": "uuid-xxx",
  "started_at": "2026-06-01T00:00:00Z",
  "expired_at": null
}
```

#### 业务规则

**分配模块：**

1. 校验模块存在且 `is_active=true`
2. 校验该租户未已分配该模块（唯一约束）
3. 创建 `TenantModule` 记录
4. 发布 `ModuleAssigned` 事件 → IAM 模块监听，创建租户实例层数据
5. 如果模块 `is_need=true`（必须模块），分配时不可设置过期时间

**取消分配：**

1. 校验模块不是必须模块（`is_need=false`），必须模块禁止取消
2. 检查租户实例层是否有用户使用该模块的角色（`UserRole` 关联）
   - 有用户使用 → 返回错误，提示先移除相关用户角色
   - 无用户使用 → 继续
3. 删除 `TenantModule` 记录
4. 发布 `ModuleUnassigned` 事件 → IAM 模块监听，级联删除租户实例层数据

**ModuleUnassigned 事件处理：**

```
1. 删除该租户所有 ref_id 指向该模块的 RolePermission
2. 删除该租户所有 ref_id 指向该模块的 Role
3. 删除该租户所有 ref_id 指向该模块的 Permission
4. 删除该租户所有 ref_id 指向该模块的 Menu（含子菜单）
5. 按依赖顺序：RolePermission → Role → Permission → Menu
```
