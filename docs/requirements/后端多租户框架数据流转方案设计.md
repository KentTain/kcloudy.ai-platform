# 插件与模型资源管理设计需求

 1.tenant模块负责插件资源的管理（插件的定义），解决有什么的问题；2.ai模块负责插件的使用（每个租户配置及使用的功能，是每个插件的实例化），解决的是用什么的问题；3.这两个是具体业务实现的地方：server\python\src\ai_plugin、server\python\src\ai\components\plugin；

## 一、插件表重新设计

### 1. plugins → 移入 Tenant schema

  当前: plugins                           → 目标: tenant.plugin_definitions

  plugin_id                          (保留)
  plugin_unique_identifier            (保留)
  refers                              (保留 — 安装计数，管理面板需要)
  install_type                        (保留)
  manifest_type                       (保留)
  remote_declaration                  (保留)

  为什么移入 Tenant？ 这是"有什么插件可用"的全局注册表，定义了插件的元数据。AI 不需要直接访问这张表——它通过安装记录获取具体配置即可。Tenant 负责管理插件的上
  架/下架。

  表名建议： plugin_definitions（比 plugins 更清晰表达"定义层"含义）

### 2. plugin_declarations → 移入 Tenant schema，或合并

  当前: plugin_declarations             → 目标: tenant.plugin_declarations (或合并入 plugin_definitions)

  plugin_id                           (保留)
  plugin_unique_identifier            (保留，唯一)
  declaration                         (保留 — 完整清单)

  是否合并？ plugin_definitions.remote_declaration 和 plugin_declarations.declaration 语义重叠。建议 合并进 plugin_definitions 加一个 declaration JSONB 字
  段。原因是这两张表都是 1:1 的插件定义数据，拆分没有实际收益。

  合并后 tenant.plugin_definitions:
    plugin_id, plugin_unique_identifier, refers, install_type,
    manifest_type, remote_declaration, declaration ← 合并

### 4. plugin_credentials → 留在 AI

  凭证是"用什么"层面的配置——租户给插件配 API Key 是使用行为。留在 AI schema，物理表不变。

### 5. plugin_install_tasks → 建议留在 AI

  安装任务进度是 AI 安装流程的运行时状态标记。但安装动作由 Tenant 触发。这个有跨度的——你觉得倾向于放哪里？

   维度                  放 Tenant             放 AI
  ━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   安装是管理动作        ✅ Tenant 触发安装    —
  ────────────────────  ────────────────────  ──────────────────────────────────
   进度是运行时状态      —                     ✅ install_task_manager.py 在 AI
  ────────────────────  ────────────────────  ──────────────────────────────────
   管理面板需要看进度    直接查同 schema       需要 inner API 跨模块查

  我倾向于留在 AI，因为 install_task_manager.py 对这张表有密集写入（进度更新）。Tenant 可以通过 inner API 查询任务状态。

## 二、Inner API 设计

  基于你的"业务模块通过 inner 接口获取数据组装"的思路：

### 接口定义位置：framework/（复用已有模式）

  已存在的模式：

  framework/tenant/protocols.py          → TenantProvider 协议
  framework/tenant/context.py            → 全局访问入口
  tenant/services/tenant_provider_impl.py → Tenant 实现

  新增：

  framework/tenant/plugin_protocols.py   → PluginInstallationProvider 协议
  tenant/services/plugin_provider.py      → Tenant 实现

### 接口方法定义

  核心是两个场景：

  场景 1：AI 初始化时批量拉取安装列表

  class PluginInstallationProvider(Protocol):
      async def get_tenant_installations(
          self, tenant_id: str
      ) -> list[PluginInstallationData]:
          """AI 启动时调用，获取租户所有已安装的插件及其能力定义"""

  @dataclass
  class PluginInstallationData:
      id: str
      plugin_id: str
      plugin_unique_identifier: str
      plugin_type: str
      runtime_type: str
      status: str                    # ACTIVE/INACTIVE
      installed_at: datetime | None
      plugin_config: dict | None     # ← 这是 AI 最需要的数据
      auto_start: bool | None

  场景 2：AI 安装/启停插件时回写状态

  class PluginInstallationProvider(Protocol):
      # Tenant 调用（安装/卸载/启停的触发器）
      async def install_plugin(
          self, tenant_id: str, plugin_id: str, ...
      ) -> PluginInstallationData: ...

      async def uninstall_plugin(
          self, tenant_id: str, installation_id: str
      ) -> None: ...

      # AI 调用（安装完成后，AI 回写启动状态）
      async def update_installation_status(
          self, installation_id: str, status: str
      ) -> None: ...

  ai/components/plugin 中的改动范围：

  plugin_manager.py:
    _load_plugins_metadata_from_database()
      session.execute(select(PluginInstallation).where(...))  →  provider.get_tenant_installations(tenant_id)

    _load_plugin_info_from_installation(installation: PluginInstallation)
      reading: installation.plugin_config, installation.status  →  从 PluginInstallationData 取值

    install_plugin() / _register_plugin_installation()
      creating PluginInstallation ORM record  →  provider.install_plugin()
      writing process_id/port to installation  →  写入 ai.plugin_runtime_states

  plugin_service.py:
      get_plugin_list() / get_plugin_info()
      ORM queries on PluginInstallation  →  provider.get_tenant_installations()

  核心业务逻辑不变，只有数据访问层替换。

  ai_plugin/ 不受影响——它是 SDK 协议层，不直接访问数据表。
  ModelClient.fetch_model_providers() 不受影响——它通过 plugin_manager.plugins 内存对象获取数据。

## 三、模型表归属

  按 "有什么/用什么" 原则：

### model_providers → 留在 AI

  model_providers (当前在 ai schema):
    provider_name    ← 来自插件定义（Tenant）
    provider_type    ← 来自插件定义（Tenant）
    plugin_id        ← 关联安装记录（Tenant）
    credentials      ← API Key — "用什么"的凭证
    is_valid         ← 软删除

  credentials 是"如何连接供应商"的配置——使用层面的。留在 AI。注意 plugin_id 会跨 schema 引用 Tenant 的 plugin_installations，应用层处理即可。

### model_configs → 留在 AI

  model_configs (当前在 ai schema):
    provider_id      ← FK → model_providers.id（同 schema，没问题）
    model_name       ← 选择哪个模型
    model_type       ← 模型类型
    parameters       ← 默认参数
    is_valid         ← 软删除

  "选择哪个模型、配什么参数"是纯使用行为。留在 AI。

  两表都不需要移动，只需要把 model_providers.plugin_id 的引用从 ai.plugin_installations 指向 tenant.plugin_installations。
