# Alon 知识库（kbhub）迁移方案

本文档描述将 Alon 项目 `apps/kbhub`（企业知识库子应用）迁移至本项目的完整方案。覆盖迁移范围、功能映射、权限分层、数据库表映射、分阶段实施计划与验收标准。

- 源项目：`D:\Project\ai\Alon\apps\kbhub`（后端 `src/kbhub`、前端 `web`、测试 `tests`、文档 `docs`）
- 目标项目：`D:\Project\kcloudy\kcloudy.ai-platform`（后端 `server/python/src`、前端 `web/vue/src`）
- 文档依据：`apps/kbhub/docs/README.md` 及 `docs/features/*`

---

## 1. 背景与目标

### 1.1 背景

kbhub 是 Alon 平台的企业知识库扩展包，承担文档资产、知识库、入库审核、权限治理、智能问答与审计合规。本项目（AI Platform）是模块化单体架构，已有 `framework`、`tenant`、`iam`、`document`、`ai`、`demo` 等模块，其中 `document` 与 `ai` 仅有模块骨架，`iam` 已具备基础 RBAC 与审计日志能力。

### 1.2 迁移目标

| 目标 | 说明 |
|------|------|
| 功能对齐 | 完整迁移 kbhub 文档库、知识库、入库审核、权限治理、人设、平台设置、工具库、站内信、审计等能力 |
| 架构适配 | 从 Alon 扩展包模式改造为本项目模块化单体（多 schema + ModuleDefinition 声明） |
| 权限分层 | IAM 负责功能权限（菜单/按钮 RBAC），各业务模块自管扩展权限（资源权限、成员权限、数据权限） |
| 复用现有 | 复用现有 IAM 审计日志、RBAC、组织/用户模型；复用 AI 模块已有 LLM/Embedding/Rerank/插件能力 |
| 前端重实现 | 使用本项目组件重新实现前端，缺少的组件从 kbhub 迁移适配 |

### 1.3 关键约束

- **ai 模块依赖 document 模块**：知识库文档来源于文档库文件，因此 document 必须先于 ai 迁移。
- **使用现有 RBAC 替换 kbhub 权限体系的功能权限部分**：kbhub 不再维护独立菜单/按钮权限，统一接入 IAM `PermissionDef`。
- **审计日志使用现有 IAM 设计**：`iam.models.audit_log.AuditLog` + `AuditLogService` + 控制器已就绪，各业务模块通过统一写入机制记录审计，不重复造表。
- **完整迁移 kbhub 权限体系的资源权限部分**：文档库资源 ACL、权限组、继承链、知识库成员角色、企业 Policy、权限申请流程完整迁移到对应业务模块。

## 3. 权限体系分层设计

### 3.1 三层权限架构

kbhub 原有一套自洽但庞大的权限体系（`services/permission.py` 约 7.8 万行）。迁移时拆为三层，分别落到不同模块：

```
┌──────────────────────────────────────────────────────────────────────┐
│  第 1 层：IAM 功能权限（RBAC，已有，完整迁移替换 kbhub 菜单/按钮权限）│
├──────────────────────────────────────────────────────────────────────┤
│  职责：菜单可见性 + 按钮可操作性                                      │
│  模型：User -> Role -> Permission -> Menu（已有）                     │
│  声明：各模块 module.py 中 PermissionDef，编码格式 {module}:{res}:{act}│
│  示例：                                                               │
│    document:library:create     创建文档库（菜单/按钮可见）            │
│    document:library:delete      删除文档库                            │
│    document:folder:write        文件夹操作                            │
│    document:tag:write           标签管理                              │
│    document:persona:write       人设管理                              │
│    ai:knowledge_base:read       知识库查看                            │
│    ai:knowledge_base:query      知识库问答                            │
│    ai:import:review             入库审核                              │
│    ai:tool:import               工具导入                              │
│    ai:platform_settings:write   平台设置                              │
│    iam:notification:read        站内信                                │
│    iam:permission_request:approve  权限申请审批                       │
│  实现：已就绪，通过 ModuleDefinition 同步到租户实例层                 │
└──────────────────────────────────────────────────────────────────────┘
                                  ⬇
┌──────────────────────────────────────────────────────────────────────┐
│  第 2 层：业务模块扩展权限（各模块自管，完整迁移 kbhub 资源/成员权限）│
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌─ document 资源权限 ─────────────────────────────────────────┐     │
│  │  职责：文档库内资源访问控制                                    │     │
│  │  模型：                                                        │     │
│  │    LibraryMember        成员固定角色 owner/admin/contributor/  │     │
│  │                         member/viewer                          │     │
│  │    LibraryRole          权限组（全员权限 + 自定义权限组）       │     │
│  │    LibraryRoleMember    权限组成员                              │     │
│  │    ResourceAcl          资源权限 none/readonly/editable        │     │
│  │    权限继承链            文档库根 -> 目录 -> 文件               │     │
│  │  判定流程：                                                    │     │
│  │    1. owner/admin 命中 -> 全部资源可编辑                       │     │
│  │    2. 计算 全员权限 + 自定义权限组 + 用户直授权               │     │
│  │    3. 沿继承链计算资源权限，取最高等级                         │     │
│  │    4. 叠加第 3 层企业 Policy                                   │     │
│  └──────────────────────────────────────────────────────────────┘     │
│                                                                      │
│  ┌─ ai 成员权限 ──────────────────────────────────────────────┐     │
│  │  职责：知识库内成员角色与操作权限                              │     │
│  │  模型：                                                        │     │
│  │    KnowledgeBaseMember  角色 owner/admin/manager/             │     │
│  │                         reviewer/query_user                   │     │
│  │    KnowledgeBaseAcl     例外授权/拒绝（不放大源文档权限）       │     │
│  │  判定流程：                                                    │     │
│  │    1. 校验知识库成员身份与操作身份                              │     │
│  │    2. 实时回查 document 源文件权限（不放大）                   │     │
│  │    3. 叠加第 3 层企业 Policy                                   │     │
│  └──────────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────────┘
                                  ⬇
┌──────────────────────────────────────────────────────────────────────┐
│  第 3 层：企业 Policy（全局策略，新增，放 iam 模块）                  │
├──────────────────────────────────────────────────────────────────────┤
│  职责：租户级安全策略（密级、下载限制、DLP、水印、入库、问答等）     │
│  规则：Policy 拒绝优先于所有允许                                     │
│  落点：iam.policies + iam.permission_cache_events（变更流水/排障）   │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 层间协作规则

```
请求进入
   │
   ▼
┌──────────────────────────────────┐
│ 第 1 层：IAM RBAC                 │  按钮能不能点、菜单能不能看
│ （功能权限，由 module 声明）       │
└──────────────────────────────────┘
   │ 通过 -> 进入业务逻辑
   ▼
┌──────────────────────────────────┐
│ 第 2 层：业务模块资源权限         │  能不能访问这个具体资源
│ document ResourceAcl / ai 成员   │
│ （资源权限，各模块自管）          │
└──────────────────────────────────┘
   │ 通过 -> 叠加策略
   ▼
┌──────────────────────────────────┐
│ 第 3 层：企业 Policy              │  全局安全策略是否拒绝
│ iam.policies                      │
└──────────────────────────────────┘
   │
   ▼
最终允许 / 拒绝 + 写审计日志（iam.audit_log）
```

### 3.3 权限申请与站内信

kbhub 的权限申请流程完整迁移到 iam 作为全局入口，审批落地动作仍回调各业务模块：

| 申请类型 | 审批范围 | 通过后落地动作 |
|----------|----------|----------------|
| library_join | 文档库管理员 | 创建 document.library_members |
| library_resource | 文档库管理员 | 创建/替换 document.resource_acls |
| library_role | 文档库管理员 | 创建 document.library_role_members |
| knowledge_base_join | 知识库管理员 | 创建 ai.knowledge_base_members |
| knowledge_base_role | 知识库管理员 | 更新 ai.knowledge_base_members.kb_role |

站内信（iam.notifications）承载审核提醒与跳转入口，但跳转后仍回源业务模块重新校验权限（消息可见 ≠ 业务可操作）。

## 5. 各模块迁移详述

### 5.1 document 模块（后端）

```
server/python/src/document/
├── module.py                     # 新建：ModuleDefinition（菜单+功能权限声明）
├── models/
│   ├── library.py                # Library / LibraryMember
│   ├── folder.py                 # Folder（TreeNodeMixin）
│   ├── document.py               # Document / DocumentVersion
│   ├── permission.py             # LibraryRole / LibraryRoleMember / ResourceAcl
│   ├── tag.py                    # Tag / TagGroup
│   ├── persona.py                # Persona
│   ├── metadata.py               # ResourceMetadata / LibraryMetadataField
│   └── enums.py                  # 文档库相关枚举（从 kbhub enums.py 拆分）
├── schemas/                      # 请求/响应 DTO
├── services/
│   ├── library_service.py        # 文档库 CRUD
│   ├── folder_service.py         # 文件夹树管理
│   ├── document_service.py       # 文件上传/预览/下载/切片
│   ├── member_service.py         # 成员管理
│   ├── permission_service.py     # 权限配置 + 资源权限判定引擎
│   ├── tag_service.py            # 标签管理
│   ├── persona_service.py        # 人设管理
│   ├── recycle_service.py        # 回收站
│   └── metadata_service.py       # 元数据管理
├── controllers/
│   ├── admin/                    # 管理端
│   ├── console/                  # 用户端
│   └── inner/                     # 内部接口（供 ai 模块调用）
├── tasks/                        # 文档切片与索引任务
├── listeners/                    # 事件监听
└── migrations/                   # 数据库迁移 + 种子
```

**关键设计点**：

1. **文档库权限判定引擎**（`permission_service.py`）：迁移 kbhub 的判定流程，但精简实现（原文件 7.8 万行含大量 Alon 平台逻辑）。判定流程见 §3.1 第 2 层。
2. **inner 接口**：供 ai 模块回查源文件权限，关键接口：
   - `GET /document/inner/v1/documents/{id}/permission?user_id=` — 校验用户对源文件的资源权限
   - `GET /document/inner/v1/libraries/{id}/members?user_id=` — 校验文档库成员身份
3. **文档切片与索引**：复用 ai 模块的 Embedding/GraphRAG 能力，document 只负责触发任务与记录状态。

### 5.2 ai 模块扩展（后端）

```
server/python/src/ai/
├── models/
│   ├── knowledge_base.py         # 新建：KnowledgeBase / Member / Document / Acl
│   ├── import_request.py         # 新建：ImportRequest / ImportRequestItem
│   ├── tool.py                   # 新建：Tool / ToolAuth / ToolFunction / ToolParameter / ImportRecord
│   └── config_item.py            # 新建：ConfigItem（平台设置）
├── services/
│   ├── knowledge_base/           # 知识库服务
│   ├── import_review/            # 入库审核流程
│   ├── tool/                     # 工具库
│   │   ├── tool_service.py       # 工具管理
│   │   ├── swagger_import.py     # Swagger 导入（prance + openapi-spec-validator）
│   │   ├── mcp_import.py         # MCP 导入（官方 SDK）
│   │   └── tool_runtime.py       # 测试 Runtime（HttpExecutor / McpExecutor）
│   └── platform_settings.py      # 平台设置
├── controllers/
│   ├── admin/                    # 知识库/工具管理端
│   ├── console/                  # 知识库问答/检索/入库
│   └── inner/
├── listeners/                    # 入库审核消息、索引任务回调
└── migrations/
```

**关键设计点**：

1. **知识库权限不放大源文档权限**：问答检索前按权限生成可访问范围，检索后对召回片段再次过滤，无权限片段不进入 LLM prompt。回查源文件权限走 document inner 接口。
2. **工具库统一 Runtime**：`ToolRuntime` 按 `Tool.protocol` 分发 `HttpExecutor`/`McpExecutor`，Connection Test 与 Function Test 共用协议分发。Swagger 用 `prance` 解析 + `openapi-spec-validator` 校验；MCP 用官方 SDK 的 SSE/Streamable HTTP。
3. **平台设置**：保存到 `ai.config_items`（`config_scope=tenant`，`config_key=kbhub_platform_settings`），含切片参数与六类模型配置。模型选择复用 ai 模块现有模型能力。
4. **入库审核**：申请提交后写 `import_requests`，通过站内信通知审核员；审核通过生成知识库文档引用。

### 5.3 iam 模块扩展（后端）

**关键设计点**：

## 8. 关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| kbhub 服务文件巨大（permission.py 7.8 万行、user.py 5.3 万行、tool.py 3.6 万行） | 迁移工作量大、易引入 Alon 平台特有逻辑 | 精简实现，只迁移核心业务逻辑；Alon 平台依赖替换为 framework 机制；按模块拆分文件 |
| 枚举定义庞大（enums.py 3.2 万行） | 维护困难 | 按模块拆分，document/ai 各自维护相关枚举，只迁移所需 |
| Alon 平台依赖（BaseExtension、alon.models.events、alon.components.scheduler/messaging） | 直接迁移不可用 | 替换为本项目 ModuleDefinition + framework 事件/调度/消息机制 |
| 前端组件依赖 Alon 特有组件（AlonLargeModel 等） | 不可直接复用 | 适配本项目模型能力与 IAM 用户/组织接口 |
| 三层权限协同复杂 | 权限判定不一致 | 统一权限判定引擎入口，Policy 拒绝优先；端到端测试覆盖（Phase 3.21） |
| 知识库放大源文档权限风险 | 越权访问 | 问答检索前后双重过滤，回查 document inner 接口，不放大权限 |
| 工具库 MCP/Swagger 外部依赖 | 导入失败、安全风险 | 校验前置、超时控制、敏感信息脱敏、仅超管可测试写操作 |
| DDL 从单 schema 拆多 schema | 迁移脚本复杂 | 按 §4 表映射逐表迁移，复用 Mixin，向量维度统一确认（默认 1024） |
| ai 依赖 document 的时序 | 若并行迁移会断链 | 严格 Phase 2 先于 Phase 3，document inner 接口先行稳定 |

## 9. 验收标准

### 9.1 通用验收

- 所有新增/修改模块的 `README.md`、`CLAUDE.md` 中文文档齐全且与实际一致。
- 后端单元测试、前端单元测试、E2E 测试全部通过。
- 数据库迁移脚本可正向执行；Ruff、Pyright、`git diff --check` 通过。
- Git 提交信息使用中文，遵循约定式提交。

### 9.2 权限体系验收

- IAM RBAC 控制菜单与按钮可见性，各模块通过 `PermissionDef` 声明并同步到租户实例层。
- document 资源权限判定：owner/admin 全可编辑；全员权限 + 自定义权限组 + 用户直授权取最高；继承链正确；Policy 拒绝优先。
- ai 知识库权限不放大源文档权限：无源文件权限时片段不进入问答上下文。
- 权限申请审批通过后，iam 通过 inner 接口回调 document/ai 正确落地授权。
- 权限排障能输出最终允许/拒绝及命中原因。

### 9.3 功能验收

- 文档库：列表/详情/文件管理/成员/权限配置/回收站/切片索引/元数据/审计完整可用。
- 标签与人设：标签 CRUD 引用人设，人设被引用时不可删除。
- 知识库：列表/详情/文档/检索测试/成员权限/问答/图谱/配置/审计完整可用。
- 入库审核：申请提交→站内信通知→审核通过→生成知识库文档引用链路通。
- 工具库：Swagger/MCP 导入、重导入、工具测试 Runtime、敏感信息脱敏符合设计。
- 平台设置：切片参数与六类模型配置可保存，影响后续新建任务。
- 站内信：消息列表/详情/跳转，跳转后回源重新校验权限。
- 审计日志：各模块操作统一写入 `iam.audit_logs`，跨模块查询一致。

### 9.4 集成验收（Phase 3 末）

- document → ai 文档引用链路端到端通。
- 三层权限端到端验证通过。
- 站内信跳转回源校验通过。
- 审计日志跨模块一致。
