## 关键设计决策确认

  在开始实现前，需要确认几个设计决策：

### 1. 数据库连接池缓存策略

  ┌─────────────────────────────────────────────────────────────────────────────────┐
  │  DatabaseEnginePool 连接池缓存策略                                              │
  ├─────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                 │
  │  方案 A：惰性加载 + LRU 回收                                                    │
  │  ├─ 首次访问租户时创建连接池                                                    │
  │  ├─ 最大缓存连接池数量限制（如 50 个）                                          │
  │  ├─ 超过限制时回收最近最少使用的连接池                                          │
  │  └─ 空闲连接池 30 分钟无访问自动回收                                            │
  │                                                                                 │
  │  方案 B：预加载 + 永久缓存                                                      │
  │  ├─ 应用启动时加载所有租户连接池                                                │
  │  ├─ 适合租户数量较少且稳定的场景                                                │
  │  └─ 缺点：内存占用高，新租户需要重启                                            │
  │                                                                                 │
  │  推荐：方案 A（惰性加载 + LRU 回收）                                            │
  │                                                                                 │
  └─────────────────────────────────────────────────────────────────────────────────┘

### 2. 默认资源处理

  ┌─────────────────────────────────────────────────────────────────────────────────┐
  │  租户未配置独立资源时的行为                                                      │
  ├─────────────────────────────────────────────────────────────────────────────────┤
  │                                                                                 │
  │  当 Tenant.db_name 为空时：                                                     │
  │  ├─ 使用默认数据库（当前共享数据库）                                            │
  │  └─ 使用 tenant_id 字段隔离（兼容现有逻辑）                                     │
  │                                                                                 │
  │  当 Tenant.storage_bucket 为空时：                                              │
  │  ├─ 使用默认 Bucket                                                            │
  │  └─ 使用 tenant_id/ 前缀路径隔离（兼容现有逻辑）                                │
  │                                                                                 │
  │  当 Tenant.cache_db 为空时：                                                    │
  │  ├─ 使用默认 Redis DB (0)                                                      │
  │  └─ 使用 {tenant_id}:key 前缀隔离（兼容现有逻辑）                               │
  │                                                                                 │
  │  优势：平滑迁移，新租户用物理隔离，旧租户继续用逻辑隔离                          │
  │                                                                                 │
  └─────────────────────────────────────────────────────────────────────────────────┘
