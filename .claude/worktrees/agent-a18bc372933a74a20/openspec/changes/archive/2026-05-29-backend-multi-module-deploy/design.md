## 上下文

后端 Python 多模块 Docker 构建及部署的最后一步，实现构建时模块选择和角色分离部署。前端模块系统已支持动态模块加载，后端菜单系统已提供菜单 API，现在需要 Docker 构建和部署配置来完成独立部署能力。

参考设计文档：`docs/designs/前后端多模块的构建及部署文档.md`

### 约束

- 后端使用 FastAPI + Uvicorn
- 使用 uv 作为包管理器
- 模块依赖：demo → iam → tenant
- 已有 application_web.py、application_task.py、application_listener.py 三个入口

## 目标 / 非目标

**目标：**

- 创建支持 BUILD_MODULES 和 APP_ROLE 参数的 Dockerfile
- 实现构建时生成 modules.py 配置文件
- 配置角色分离部署（web/task/listener）
- 定义平台版和独立模块版的 docker-compose 服务
- 更新部署文档和开发手册

**非目标：**

- Kubernetes 部署配置
- CI/CD 流水线配置
- 生产环境安全加固（HTTPS、安全头等）

## 决策

### 决策 1：构建时模块选择 vs 运行时模块加载

**选择：** 构建时模块选择

**理由：**
- 与前端方案保持一致
- 镜像语义清晰，标签显示包含的模块
- 自动依赖解析保证正确性
- 实现简单，只需小改动

**替代方案：**
- 运行时模块加载 → 增加复杂度，需要模块服务器
- 混合方案 → 增加维护成本

### 决策 2：角色分离策略

**选择：** 独立容器部署

**理由：**
- 职责单一，便于监控和日志管理
- 可独立扩展和重启
- 故障隔离，一个角色崩溃不影响其他角色

**替代方案：**
- 单容器多进程 → 简单但不够灵活
- Supervisor 管理 → 增加依赖

### 决策 3：依赖解析策略

**选择：** 自动解析依赖

**理由：**
- 用户无需理解模块依赖关系
- 利用现有的 `load_modules()` 函数
- 保证依赖完整性

**示例：**
- BUILD_MODULES=demo → 自动包含 demo, iam, tenant
- BUILD_MODULES=iam → 自动包含 iam, tenant
- BUILD_MODULES=tenant → 只包含 tenant

### 决策 4：配置文件生成

**选择：** 构建时生成 Python 配置文件

**理由：**
- 类型安全，Python 编译时检查
- 与应用代码无缝集成
- 简单直接，无需额外解析

### 决策 5：Entrypoint 设计

**选择：** Shell 脚本根据 APP_ROLE 启动

**理由：**
- 灵活，易于调试
- 支持环境变量覆盖
- 可添加启动前钩子（如数据库迁移）

## 风险 / 缓解

| 风险 | 缓解措施 |
|------|----------|
| 构建参数错误导致模块缺失 | 验证 BUILD_MODULES 格式，提供默认值 |
| 依赖解析循环依赖 | 利用现有 `resolve_dependencies()` 检测 |
| 数据库迁移冲突 | 统一迁移入口脚本，按模块顺序执行 |
| 配置文件路径错误 | 使用绝对路径，添加存在性检查 |
| Task/Listener 启动失败 | 添加健康检查，失败时重启 |
