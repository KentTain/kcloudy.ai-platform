## 上下文

设计文档 `docs/requirements/9.Tenant模块的系统设计.md` 已完整规划租户模块化架构。当前系统缺少：
- 资源配置独立管理能力（Database/Storage/Cache/Queue/PubSub）
- 模块定义层（菜单/权限/角色的模板定义）
- 租户模块分配与自动同步机制

约束：
- 必须遵循三层架构（Controller → Service → Model）
- Tenant 模块发布事件，IAM 模块监听处理（跨模块职责分离）
- 租户实例层数据由同步机制维护，禁止租户直接修改

## 目标 / 非目标

**目标：**
- 实现五类资源配置的 CRUD 及连通性测试
- 实现模块定义层模型及 API
- 实现租户模块分配与自动同步（菜单/权限/角色）
- 通过领域事件实现 Tenant → IAM 的跨模块通信

**非目标：**
- 不实现前端界面（仅后端 API）
- 不实现租户数据隔离的动态切换（已有 `tenant-database-isolation` 等规范）
- 不实现模块的版本管理与灰度发布

## 决策

### D1: 资源配置独立于租户

**决策：** 资源配置（Database/Storage 等）作为独立实体，多个租户可共享同一配置。

**理由：** 小型租户通常共享基础设施，独立配置支持资源复用，降低运维成本。租户通过 FK 关联配置，支持一对一绑定。

**替代方案：**
- 配置内嵌于租户（拒绝理由：无法复用，配置冗余）
- 配置作为租户子资源（拒绝理由：语义模糊，租户删除时配置无法保留）

### D2: 模块定义层与租户实例层分离

**决策：** 模块定义层（Module、ModuleMenu 等）存储全局模板，租户实例层（Menu、Permission 等）通过 `ref_id` 关联模板。

**理由：**
- 模块定义可复用，多个租户分配同一模块时共享模板
- 模块定义变更时，可通过事件同步到所有已分配租户
- 租户无法定制菜单/权限，确保一致性

**替代方案：**
- 直接在租户实例层创建菜单/权限（拒绝理由：无模板，无法统一管理）
- 租户继承模块定义后允许定制（拒绝理由：违背"纯镜像"原则，增加复杂度）

### D3: 事件驱动同步机制

**决策：** Tenant 模块发布领域事件（ModuleAssigned、ModuleMenuCreated 等），IAM 模块监听事件并执行同步。

**理由：**
- 职责分离：Tenant 模块专注模块分配，IAM 模块专注用户权限
- 解耦：Tenant 不依赖 IAM 的内部实现
- 可扩展：未来可增加其他监听器（如计费、审计）

**替代方案：**
- Tenant 模块直接调用 IAM API（拒绝理由：违反模块边界，增加耦合）
- 定时任务轮询同步（拒绝理由：延迟高，资源浪费）

### D4: 默认角色设计

**决策：** 每个模块定义两个系统角色（`is_system=true`）：系统管理员（全权限）、普通用户（只读）。

**理由：** 简化初始权限配置，覆盖常见场景。租户可基于系统角色创建自定义角色。

**替代方案：**
- 不提供默认角色（拒绝理由：租户需要手动配置基础角色，体验差）
- 提供细粒度默认角色（拒绝理由：过度设计，增加理解成本）

### D5: 取消模块分配的保护机制

**决策：** 取消模块分配前检查是否有用户使用该模块的角色，有则拒绝并提示。

**理由：** 防止误操作导致用户权限丢失，提供明确的错误提示引导管理员处理。

**替代方案：**
- 强制取消并级联删除 UserRole（拒绝理由：用户无感知，风险高）
- 软删除保留数据（拒绝理由：数据冗余，清理逻辑复杂）

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 事件丢失导致数据不一致 | 事件持久化 + 重试机制；提供手动同步 API |
| 大量租户同步时性能问题 | 批量处理 + 异步队列；限制单模块租户数 |
| 密码存储安全 | AES 加密存储，读取时脱敏；密钥轮换机制 |
| 资源配置删除影响租户 | FK 约束禁止删除已引用配置；返回明确错误提示 |

## 迁移计划

### 阶段 1：数据库迁移

1. 创建 5 张资源配置表
2. 创建 TenantConfig 表
3. 修改 Tenant 表（新增 5 个 FK 字段）
4. 创建 Module 相关 5 张表
5. 创建 TenantModule 表
6. 修改 IAM 模块 6 张表（新增 ref_id、tenant_id）

### 阶段 2：后端实现

1. 实现资源配置 Service/Controller
2. 实现模块定义 Service/Controller
3. 实现租户资源绑定 API
4. 实现租户模块分配 API
5. 实现事件发布机制
6. 实现 IAM 同步处理器

### 阶段 3：测试验证

1. 资源配置连通性测试
2. 模块分配同步测试
3. 取消分配保护测试
4. 事件处理幂等性测试
