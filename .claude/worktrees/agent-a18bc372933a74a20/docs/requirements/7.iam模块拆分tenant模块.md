# IAM 模块拆分 Tenant 模块方案

## 概述

本文档描述将 IAM 模块中的租户相关功能拆分为独立 Tenant 模块的方案，以及引入内部接口（inner）层的模块间通信设计。

## 背景与动机

### 当前架构问题

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         当前 IAM 模块职责过重                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  IAM (Identity and Access Management) 应该聚焦于：                          │
│  ├── 用户身份：认证、用户信息                                                │
│  ├── 访问控制：角色、权限（RBAC）                                            │
│  └── 组织架构：部门、组织关系                                                │
│                                                                             │
│  Tenant（租户）本质上是：                                                    │
│  ├── 多租户系统的核心概念                                                    │
│  ├── 资源隔离的边界                                                          │
│  └── 业务数据的分组容器                                                      │
│                                                                             │
│  Tenant 不是 IAM 特有的概念，它是整个系统的基础设施概念                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 拆分目标

1. **职责清晰**：Tenant 作为独立的基础设施模块，IAM 聚焦身份与访问管理
2. **依赖合理**：业务模块依赖 Tenant 模块获取租户上下文
3. **支持独立部署**：Tenant 模块可独立部署，供多个业务系统共享

### 1.2 模型归属决策

| 模型 | 归属模块 | 决策理由 |
|------|----------|----------|
| `tenant` | tenant | 租户核心实体，独立于 IAM |
| `tenant_config` | tenant | 租户资源配置，与 tenant 一对一 |
| `tenant_admin` | tenant | 全局管理员，不属于任何租户 |
| `user_tenant` | iam | 用户-租户关联，与 User 强耦合 |

## 二、接口分层设计

### 2.1 四层接口架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          四层接口架构                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  /admin/v1/*          管理后台接口                                    │   │
│  │  ├── 面向：管理后台前端                                                │   │
│  │  ├── 权限：全局/租户管理员                                             │   │
│  │  └── 示例：创建租户、配置资源、管理系统设置                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  /console/v1/*        用户控制台接口                                  │   │
│  │  ├── 面向：用户端前端                                                  │   │
│  │  ├── 权限：登录用户                                                    │   │
│  │  ├── 特点：用户视角，包含用户上下文                                     │   │
│  │  └── 示例：获取当前租户、切换租户                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  /api/v1/*            通用业务接口                                    │   │
│  │  ├── 面向：前端、可能的第三方集成                                      │   │
│  │  ├── 权限：登录用户                                                    │   │
│  │  ├── 特点：RESTful 风格，标准 CRUD                                     │   │
│  │  └── 示例：/api/v1/users, /api/v1/auth                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  /inner/v1/*          内部接口（新增）                                │   │
│  │  ├── 面向：模块间调用                                                  │   │
│  │  ├── 权限：内部可信，或服务间认证                                      │   │
│  │  ├── 特点：仅供模块间调用，不对外暴露                                   │   │
│  │  └── 示例：/inner/v1/users/{id}, /inner/v1/tenants/{id}               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 console 与 inner 的区别

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  console（用户控制台）:                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  GET /console/v1/tenants                                             │   │
│  │  → 返回「当前用户」所属的租户列表                                      │   │
│  │  → user_id 从 Token 获取                                             │   │
│  │  → 前端调用，渲染租户选择器                                           │   │
│  │  → 语义：用户在控制台能做什么                                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  inner（内部接口）:                                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  GET /inner/v1/users/{user_id}                                       │   │
│  │  → 返回指定用户的详细信息                                              │   │
│  │  → user_id 由调用方传入                                               │   │
│  │  → demo 模块调用，获取创建者信息                                       │   │
│  │  → 语义：模块间数据交换                                                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  关键区别：                                                                  │
│  ├── console 面向前端用户，inner 面向其他模块                               │
│  ├── console 依赖 Token 上下文，inner 参数显式传递                          │
│  ├── console 返回用户视角数据，inner 返回完整/原始数据                      │
│  └── 两者不应合并，语义完全不同                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 inner 接口设计原则

| 原则 | 说明 |
|------|------|
| 不走网关暴露 | 只在内部网络可访问，生产环境通过网关配置禁止外网访问 |
| 认证方式不同 | 单体模式无需认证（同进程）；微服务模式使用服务间认证 |
| 参数显式传递 | 不依赖 Token，由调用方显式传入所需参数 |
| 支持批量操作 | 优化模块间调用效率，如 `POST /inner/v1/users/batch` |

## 三、inner 接口混合方案

### 3.1 方案概述

混合方案同时支持单体模式和微服务模式，通过配置切换调用方式。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          混合方案架构                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  调用方（如 demo 模块）                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  # demo/clients/iam_client.py                                        │   │
│  │                                                                       │   │
│  │  class IamClient:                                                    │   │
│  │      @staticmethod                                                   │   │
│  │      async def get_user(user_id: str) -> UserInfo:                  │   │
│  │          if settings.IAM_INNER_URL:  # 微服务模式                     │   │
│  │              return await InnerHttpClient.get(                       │   │
│  │                  f"{settings.IAM_INNER_URL}/users/{user_id}"        │   │
│  │              )                                                       │   │
│  │          else:  # 单体模式                                            │   │
│  │              from iam.services import UserService                    │   │
│  │              return await UserService.get_by_id(user_id)            │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  被调用方（如 iam 模块）                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  # iam/controllers/inner/user_controller.py                          │   │
│  │                                                                       │   │
│  │  router = APIRouter(prefix="/inner/v1", tags=["Inner - User"])      │   │
│  │                                                                       │   │
│  │  @router.get("/users/{user_id}")                                     │   │
│  │  async def get_user_inner(user_id: str) -> UserInfo:                │   │
│  │      return await UserService.get_by_id(user_id)                     │   │
│  │                                                                       │   │
│  │  @router.post("/users/batch")                                        │   │
│  │  async def get_users_batch(user_ids: list[str]) -> list[UserInfo]:  │   │
│  │      return await UserService.get_by_ids(user_ids)                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 配置管理

```yaml
# config/application.yml

# 单体模式（默认）
# 不配置 IAM_INNER_URL，使用本地 Service 调用

# 微服务模式
iam:
  inner_url: "http://iam-service:8001"  # 配置后启用 HTTP 调用
  inner_timeout: 30

tenant:
  inner_url: "http://tenant-service:8002"
  inner_timeout: 30
```

### 3.3 inner 接口示例

#### Tenant 模块 inner 接口

```python
# tenant/controllers/inner/tenant_controller.py

router = APIRouter(prefix="/inner/v1", tags=["Inner - Tenant"])

@router.get("/tenants/{tenant_id}")
async def get_tenant_inner(tenant_id: str) -> TenantInfo:
    """获取租户信息（内部调用）"""
    return await TenantService.get_by_id(tenant_id)

@router.post("/tenants/batch")
async def get_tenants_batch(tenant_ids: list[str]) -> list[TenantInfo]:
    """批量获取租户信息"""
    return await TenantService.get_by_ids(tenant_ids)

@router.get("/tenants/{tenant_id}/validate")
async def validate_tenant_access(tenant_id: str, user_id: str) -> bool:
    """验证用户是否有权访问租户"""
    return await TenantService.validate_access(user_id, tenant_id)
```

#### IAM 模块 inner 接口

```python
# iam/controllers/inner/user_controller.py

router = APIRouter(prefix="/inner/v1", tags=["Inner - User"])

@router.get("/users/{user_id}")
async def get_user_inner(user_id: str) -> UserInfo:
    """获取用户信息（内部调用）"""
    return await UserService.get_by_id(user_id)

@router.post("/users/batch")
async def get_users_batch(user_ids: list[str]) -> list[UserInfo]:
    """批量获取用户信息"""
    return await UserService.get_by_ids(user_ids)

@router.get("/users/{user_id}/departments")
async def get_user_departments(user_id: str) -> list[DepartmentInfo]:
    """获取用户所属部门"""
    return await DepartmentService.get_user_departments(user_id)
```

| 步骤 | 内容 | 依赖 |
|------|------|------|
| 1 | 创建 `src/tenant/` 模块目录结构 | - |
| 2 | 迁移 Tenant 相关模型到新模块 | 步骤 1 |
| 3 | 迁移 Tenant 相关服务到新模块 | 步骤 2 |
| 4 | 迁移 Tenant 相关控制器到新模块 | 步骤 3 |
| 5 | 创建 Tenant 模块迁移文件 | 步骤 2 |
| 6 | 更新 IAM 模块，移除 Tenant 相关内容 | 步骤 2-5 |
| 7 | 创建 IAM 模块迁移文件（更新 user_tenants） | 步骤 6 |
| 8 | 添加 inner 接口层到 tenant 和 iam | 步骤 4-7 |
| 9 | 创建 `InnerHttpClient` 和模块 Client 封装 | 步骤 8 |
| 10 | 前端拆分 tenant 模块 | 步骤 4 |
| 11 | 更新测试用例 | 步骤 1-10 |
| 12 | 重建迁移脚本执行 | 步骤 5, 7 |

### 6.2 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 迁移数据丢失 | 先备份数据，使用重建脚本在测试环境验证 |
| 模块间调用失败 | inner 接口添加健康检查，Client 添加重试机制 |
| 前端路由冲突 | 路由拆分后统一在 framework 注册 |
| 测试覆盖不足 | 拆分前补充集成测试，确保功能回归 |
