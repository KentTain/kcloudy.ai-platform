# IAM 模块设计需求

Identity and Access Management（身份与访问管理）模块，涵盖租户管理、用户管理、组织架构、角色权限、认证授权等核心功能。

## 1. 概述

### 1.1 模块定位

IAM 模块是系统的核心基础设施，负责：

- **身份管理**：用户、租户、组织的创建、维护与关联
- **访问管理**：认证（登录/登出）、授权（角色/权限）
- **安全控制**：密码策略、Token 管理、审计日志

### 1.2 目标用户

| 角色 | 职责 | 核心场景 |
|------|------|----------|
| 租户管理员 | 创建租户、管理租户级系统管理员 | 系统初始化时创建，负责租户开通 |
| 系统管理员 | 管理本租户的组织架构、用户、角色、权限 | 租户内的管理操作 |
| 普通用户 | 使用系统业务功能 | 日常登录、个人信息管理 |

### 1.3 核心场景

```
┌─────────────────────────────────────────────────────────────────┐
│                        IAM 核心场景                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  系统初始化                                                      │
│  └── 创建默认租户管理员                                          │
│      └── 租户管理员登录 → 创建租户 → 创建该租户的系统管理员        │
│          └── 系统管理员登录 → 创建部门/用户/角色 → 分配权限        │
│              └── 普通用户登录 → 使用业务功能                      │
│                                                                 │
│  用户登录                                                        │
│  ├── 本地登录：用户名 + 密码                                     │
│  └── OAuth2 登录：微信/企微授权 → 自动注册/绑定 → 登录成功        │
│                                                                 │
│  会话管理                                                        │
│  ├── Token 刷新：Access Token 过期 → Refresh Token 换取新 Token  │
│  └── 登出：清除会话、Token 失效                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 功能需求

### 2.1 用户注册

**需求描述**：支持用户自主注册，注册后关联默认租户。

**功能要点**：

- 用户名（唯一）、邮箱/手机号、密码
- 密码强度校验：长度 8-32 位，包含字母和数字
- 邮箱/手机号验证（可选，MVP 阶段可跳过）
- 注册成功后自动登录

**API 设计**：

- `POST /api/v1/iam/register` - 用户注册

### 2.2 本地登录

**需求描述**：通过用户名/邮箱/手机号 + 密码进行登录。

**功能要点**：

- 支持用户名、邮箱、手机号三种登录方式
- 密码安全：前端明文传输（依赖 HTTPS），后端 BCrypt 哈希存储
- 模糊提示：登录失败返回统一提示"用户名或密码错误"
- 登录限流：单 IP 每分钟最多 5 次尝试
- 登录成功后签发双 Token（Access Token + Refresh Token）

**密码存储策略**：

- 使用 BCrypt 算法，cost factor = 12
- BCrypt 自带盐值，无需额外存储
- 存储字段：`password_hash`

**API 设计**：

- `POST /api/v1/iam/login` - 用户登录
- 请求体：`{ "account": "username/email/phone", "password": "明文密码" }`
- 响应体：`{ "access_token": "...", "refresh_token": "...", "expires_in": 7200 }`

### 2.3 第三方 OAuth2 登录

**需求描述**：支持微信、企微 OAuth2 授权登录。

**功能要点**：

- 配置管理：在后台配置 client_id、client_secret、回调地址
- 授权流程：
  1. 前端引导用户跳转第三方授权页面
  2. 用户同意授权，第三方携带 code 回调
  3. 后端使用 code 换取 access_token
  4. 使用 access_token 获取用户信息（openid、昵称、头像）
  5. 查询本地数据库：已存在则登录，不存在则自动注册并绑定
- 账号关联策略：用户表 + OAuth 关联表（支持一个用户绑定多个第三方账号）

**OAuth 关联表设计**：

```sql
CREATE TABLE oauth_connections (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id),
    provider VARCHAR(20) NOT NULL,  -- wechat, wework
    provider_user_id VARCHAR(100) NOT NULL,  -- openid / unionid
    access_token TEXT,
    refresh_token TEXT,
    expires_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(provider, provider_user_id)
);
```

**实现优先级**：

1. 微信（面向个人用户）
2. 企微（面向企业用户）

**API 设计**：

- `GET /api/v1/iam/oauth/{provider}` - 获取 OAuth 授权链接
- `GET /api/v1/iam/oauth/{provider}/callback` - OAuth 回调处理
- `POST /api/v1/iam/oauth/complete-profile` - 补全用户信息

**信息补全**：

第三方登录创建的用户缺少密码、邮箱、手机号等信息，需要引导用户补全：

```
┌─────────────────────────────────────────────────────────────────┐
│                    OAuth 用户信息补全流程                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  OAuth 登录成功                                                  │
│      │                                                          │
│      ▼                                                          │
│  检查用户信息完整度                                              │
│  • password_hash 为空？                                         │
│  • email 为空？                                                 │
│  • phone 为空？                                                 │
│      │                                                          │
│      ├── 全部完整 → 正常登录，返回 Token                         │
│      │                                                          │
│      └── 信息缺失 → 返回 Token + need_complete_profile: true    │
│                        │                                        │
│                        ▼                                        │
│                   前端引导补全页面                                │
│                   • 设置密码（必填）                              │
│                   • 绑定邮箱（选填）                              │
│                   • 绑定手机（选填）                              │
│                        │                                        │
│                        ▼                                        │
│                   调用 complete-profile API                      │
│                        │                                        │
│                        ▼                                        │
│                   更新用户信息，标记已补全                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**用户表扩展字段**：

```sql
-- 在 users 表添加信息完整度标记
ALTER TABLE users ADD COLUMN profile_completed BOOLEAN DEFAULT TRUE;

-- OAuth 登录创建的用户默认 profile_completed = FALSE
```

**补全信息接口**：

- 请求体：
  ```json
  {
    "password": "新密码",
    "email": "user@example.com",      // 可选
    "phone": "13800138000",           // 可选
    "email_code": "123456",           // 邮箱验证码（如填了邮箱）
    "phone_code": "123456"            // 手机验证码（如填了手机）
  }
  ```
- 响应体：
  ```json
  {
    "success": true,
    "message": "信息补全成功"
  }
  ```

### 2.4 登出与会话管理

**需求描述**：支持用户登出，清除会话状态。

**功能要点**：

- 登出流程：
  1. 验证 Access Token 有效性
  2. 将 Token 加入 Redis 黑名单（过期时间 = Token 剩余有效期）
  3. 清除 Redis 中的会话信息
  4. 返回成功响应
- 多设备管理（迭代功能）：
  - 查看当前用户所有活跃会话
  - 踢出指定设备

**API 设计**：

- `POST /api/v1/iam/logout` - 用户登出
- `GET /api/v1/iam/sessions` - 获取活跃会话列表（迭代）
- `DELETE /api/v1/iam/sessions/{session_id}` - 踢出指定会话（迭代）

### 2.5 Token 管理

**需求描述**：JWT 双 Token 机制 + Redis 混合模式。

**功能要点**：

**Token 类型**：

| Token 类型 | 有效期 | 用途 |
|------------|--------|------|
| Access Token | 2 小时 | 业务接口鉴权 |
| Refresh Token | 7 天 | 刷新 Access Token |

**混合模式设计**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    JWT + Redis 混合模式                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Access Token 结构：                                            │
│  {                                                              │
│    "sub": "user_id",                                            │
│    "session_id": "uuid",                                        │
│    "version": 1,                                                │
│    "roles": ["admin", "user"],                                  │
│    "permissions": ["user:read", "user:write"],                  │
│    "exp": 1716182400                                            │
│  }                                                              │
│                                                                 │
│  Redis 会话存储：                                                │
│  Key: session:{session_id}                                      │
│  Value: {                                                       │
│    "user_id": "uuid",                                           │
│    "version": 1,                                                │
│    "tenant_id": "uuid",                                         │
│    "device_info": "Chrome/Windows",                             │
│    "ip": "192.168.1.1",                                         │
│    "created_at": "2024-05-20T10:00:00Z"                         │
│  }                                                              │
│  TTL: 7 天                                                       │
│                                                                 │
│  验证流程：                                                      │
│  1. 解析 Token，获取 session_id 和 version                      │
│  2. version 匹配 → 直接使用 Token 中的权限信息                   │
│  3. version 不匹配 → 查 Redis 更新权限，返回新 Token             │
│  4. Token 在黑名单 → 拒绝访问                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Token 刷新流程**：

1. Access Token 过期，前端使用 Refresh Token 请求刷新
2. 验证 Refresh Token 有效性
3. 查询 Redis 会话状态，确认未被撤销
4. 签发新的 Access Token 和 Refresh Token
5. 旧的 Refresh Token 失效

**API 设计**：

- `POST /api/v1/iam/token/refresh` - 刷新 Token
- 请求体：`{ "refresh_token": "..." }`
- 响应体：`{ "access_token": "...", "refresh_token": "...", "expires_in": 7200 }`

### 2.6 用户信息管理

**需求描述**：获取和修改当前登录用户信息。

**功能要点**：

- 获取当前用户信息：从 Token 解析 user_id，查询数据库返回
- 修改基本信息：昵称、头像、邮箱、手机号
- 修改密码：验证旧密码，设置新密码

**API 设计**：

- `GET /api/v1/iam/user/me` - 获取当前用户信息
- `PUT /api/v1/iam/user/me` - 修改当前用户信息
- `PUT /api/v1/iam/user/password` - 修改密码

### 2.7 密码重置

**需求描述**：用户忘记密码时，通过邮箱/手机号重置密码。

**功能要点**：

- 发送验证码：邮箱或手机号接收 6 位验证码，有效期 5 分钟
- 验证码限流：同一账号每分钟最多 1 次，每天最多 5 次
- 重置密码：验证码正确后，设置新密码
- 重置后自动登出所有设备

**API 设计**：

- `POST /api/v1/iam/password/reset-code` - 发送重置验证码
- `POST /api/v1/iam/password/reset` - 重置密码

## 3. 非功能需求

### 3.1 安全设计

**密码存储**：

- 算法：BCrypt，cost factor = 12
- 前端传输：明文（依赖 HTTPS 加密）
- 后端存储：BCrypt 哈希值

**传输安全**：

- 生产环境必须使用 HTTPS
- 敏感接口（登录、注册、密码重置）强制 HTTPS

**接口限流**：

| 接口 | 限流策略 |
|------|----------|
| 登录 | 单 IP 每分钟 5 次 |
| 注册 | 单 IP 每小时 10 次 |
| 密码重置 | 单账号每分钟 1 次，每天 5 次 |
| Token 刷新 | 单 Token 每分钟 10 次 |

**模糊提示**：

- 登录失败返回统一提示"用户名或密码错误"
- 不区分"用户不存在"和"密码错误"

### 3.2 多因素认证 (MFA)

**需求描述**：为高权限账号提供多因素认证支持。

**功能要点**（迭代功能）：

- 支持方式：邮箱验证码、手机验证码
- 触发场景：
  - 租户管理员、系统管理员登录
  - 异地登录检测
  - 敏感操作（删除用户、修改权限）

### 3.3 权限控制 (RBAC)

**需求描述**：基于角色的访问控制，支持租户内权限隔离。

**角色定义**：

| 角色 | 编码 | 说明 | 权限范围 |
|------|------|------|----------|
| 租户管理员 | `tenant_admin` | 系统最高权限 | 创建/管理租户、创建租户的系统管理员 |
| 系统管理员 | `system_admin` | 租户内最高权限 | 管理本租户的部门、用户、角色、权限 |
| 普通用户 | `user` | 业务用户 | 使用系统功能 |

**权限模型**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      RBAC 权限模型                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  User ──N:M──▶ Role ──N:M──▶ Permission                        │
│                                                                 │
│  用户通过角色获得权限：                                          │
│  • 一个用户可拥有多个角色                                        │
│  • 一个角色可包含多个权限                                        │
│  • 权限 = 资源 + 操作                                           │
│                                                                 │
│  权限命名规范：                                                  │
│  • 资源:操作，如 user:read, user:write, user:delete             │
│  • 支持通配符：user:* 表示用户的所有权限                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**权限数据表**：

```sql
-- 角色表
CREATE TABLE roles (
    id UUID PRIMARY KEY,
    tenant_id UUID REFERENCES tenants(id),  -- NULL 表示全局角色
    code VARCHAR(50) NOT NULL,              -- tenant_admin, system_admin, user
    name VARCHAR(100) NOT NULL,             -- 租户管理员, 系统管理员, 普通用户
    description TEXT,
    is_system BOOLEAN DEFAULT FALSE,        -- 系统内置角色不可删除
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(tenant_id, code)
);

-- 权限表
CREATE TABLE permissions (
    id UUID PRIMARY KEY,
    code VARCHAR(100) UNIQUE NOT NULL,      -- user:read, user:write
    name VARCHAR(100) NOT NULL,             -- 查看用户, 编辑用户
    resource VARCHAR(50) NOT NULL,          -- user, role, department
    action VARCHAR(20) NOT NULL,            -- read, write, delete
    description TEXT
);

-- 用户-角色关联表
CREATE TABLE user_roles (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id),
    role_id UUID NOT NULL REFERENCES roles(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, role_id)
);

-- 角色-权限关联表
CREATE TABLE role_permissions (
    id UUID PRIMARY KEY,
    role_id UUID NOT NULL REFERENCES roles(id),
    permission_id UUID NOT NULL REFERENCES permissions(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(role_id, permission_id)
);
```

**系统初始化**：

1. 系统启动时创建默认租户管理员账号
2. 租户管理员创建租户时，自动创建该租户的"系统管理员"和"普通用户"角色
3. 租户管理员为租户创建系统管理员账号

### 3.4 审计日志

**需求描述**：记录用户登录行为和关键操作。

**功能要点**：

- 登录日志：
  - 登录时间、IP、设备指纹、登录方式、是否成功
  - 异地登录检测与告警
- 操作日志：
  - 操作时间、用户、操作类型、操作对象、变更内容
  - 关键操作：创建/删除用户、修改权限、修改角色

**数据表设计**：

```sql
-- 登录日志
CREATE TABLE login_logs (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    login_type VARCHAR(20),         -- password, wechat, wework
    ip_address VARCHAR(45),
    user_agent TEXT,
    device_fingerprint VARCHAR(100),
    is_success BOOLEAN,
    fail_reason VARCHAR(200),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 操作日志
CREATE TABLE operation_logs (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    tenant_id UUID REFERENCES tenants(id),
    action VARCHAR(50),             -- user.create, user.delete, role.update
    resource_type VARCHAR(50),      -- user, role, department
    resource_id UUID,
    old_value JSONB,
    new_value JSONB,
    ip_address VARCHAR(45),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 3.5 管理后台

**需求描述**：独立的管理后台，支持跨租户管理。

**功能要点**：

- 独立认证体系（超级管理员）
- 不经过租户中间件
- 支持跨租户查询与操作

**实现方式**：

- 管理后台使用独立的 API 前缀：`/admin/v1/`
- 使用独立的认证中间件（AdminAuthMiddleware）
- 租户管理员账号存储在 `tenant_admins` 表，与业务用户隔离

## 4. 技术设计

### 4.1 数据模型

**模块位置**：`server/python/src/demo/models/iam/`

**数据表清单**：

| 表名 | 说明 | 来源 |
|------|------|------|

| `tenant_admins` | 租户管理员表 | 从 tenant 模块迁移 |
| `user_tenants` | 用户-租户关联表 | 从 tenant 模块迁移 |
| `users` | 用户表 | 新增 |
| `user_profiles` | 用户资料表 | 新增（可选，存扩展信息） |
| `oauth_connections` | OAuth 关联表 | 新增 |
| `departments` | 部门表 | 新增 |
| `roles` | 角色表 | 新增 |

| `login_logs` | 登录日志表 | 新增 |
| `operation_logs` | 操作日志表 | 新增 |

**核心表设计**：

```sql
-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(128) UNIQUE,
    phone VARCHAR(20) UNIQUE,
    password_hash VARCHAR(255),              -- OAuth 用户可为空，补全后设置
    nickname VARCHAR(100),
    avatar VARCHAR(500),
    status VARCHAR(20) DEFAULT 'active',     -- active, inactive, locked
    profile_completed BOOLEAN DEFAULT TRUE,  -- OAuth 用户首次登录为 FALSE
    is_email_verified BOOLEAN DEFAULT FALSE,
    is_phone_verified BOOLEAN DEFAULT FALSE,
    last_login_at TIMESTAMP,
    last_login_ip VARCHAR(45),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX ix_users_username ON users(username);
CREATE INDEX ix_users_email ON users(email);
CREATE INDEX ix_users_phone ON users(phone);
CREATE INDEX ix_users_status ON users(status);

-- 部门表
CREATE TABLE departments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    parent_id UUID REFERENCES departments(id),
    name VARCHAR(100) NOT NULL,
    code VARCHAR(50),                        -- 部门编码
    sort_order INTEGER DEFAULT 0,
    leader_id UUID REFERENCES users(id),     -- 部门负责人
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX ix_departments_tenant_id ON departments(tenant_id);
CREATE INDEX ix_departments_parent_id ON departments(parent_id);

-- 用户-部门关联表
CREATE TABLE user_departments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id),
    department_id UUID NOT NULL REFERENCES departments(id),
    is_leader BOOLEAN DEFAULT FALSE,         -- 是否部门负责人
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, department_id)
```

### 4.2 加密/解密工具

**位置**：`server/python/src/framework/utils/crypto.py`

**功能清单**：

| 函数 | 用途 |
|------|------|
| `hash_password(password: str) -> str` | BCrypt 密码哈希 |
| `verify_password(password: str, hash: str) -> bool` | 密码验证 |
| `generate_token() -> str` | 生成随机 Token |
| `generate_verification_code(length: int = 6) -> str` | 生成验证码 |

**实现示例**：

```python
import bcrypt

def hash_password(password: str) -> str:
    """BCrypt 密码哈希"""
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(password.encode(), salt).decode()

def verify_password(password: str, hash: str) -> bool:
    """密码验证"""
    return bcrypt.checkpw(password.encode(), hash.encode())
```

### 4.4 前端集成

**Token 存储**：

- Access Token：`localStorage` 或 `sessionStorage`
- Refresh Token：`httpOnly Cookie`（更安全，防 XSS）

**请求拦截**：

- 请求头携带：`Authorization: Bearer {access_token}`
- 401 响应：尝试刷新 Token，刷新失败跳转登录页

**路由守卫**：

- 白名单路由：登录页、注册页、OAuth 回调页
- 其他路由：检查 Token，无 Token 跳转登录页
