# 规范合规审查

## 铁律

任何实现必须与 `openspec/specs/` 中的规范一致。不一致就是 bug。

## 审查步骤

1. **读取主规范** — 检查 `openspec/specs/` 下的主规范，确认本次实现是否违反了已有需求

2. **读取增量规范** — 读取本次变更对应的 `openspec/changes/` 下的 delta 规范

3. **逐条检查场景** — 检查每个"假设/当/则"场景是否有对应实现

4. **检查架构决策** — 检查 `design.md` 中提到的架构决策是否被遵守

5. **检查排除范围** — 检查 `proposal.md` 的排除范围是否被违反

## 检查清单

为每个检查项创建任务并逐一完成：

### 1. 规范覆盖检查

- [ ] 读取 `openspec/specs/` 下的相关规范文件
- [ ] 列出所有需求（标记为 `### Requirement:` 或 `## 需求：` 的部分）
- [ ] 每个需求在代码中找到对应实现
- [ ] 记录未覆盖的需求

### 2. 场景覆盖检查

对于每个场景（标记为 `#### Scenario:` 或 `### 场景：`）：

- [ ] 正常路径有对应实现
- [ ] 错误路径有对应处理
- [ ] 边界情况有覆盖

### 3. 设计决策检查

从 `design.md` 提取关键决策：

- [ ] 技术选型决策被遵循
- [ ] 架构模式被采用
- [ ] 数据结构设计一致
- [ ] API 设计一致

### 4. 排除范围检查

从 `proposal.md` 提取排除范围：

- [ ] 未实现排除范围内的功能
- [ ] 未修改排除范围内的文件
- [ ] 未添加排除范围内的依赖

## 输出格式

```
## 规范合规审查报告

### 检查摘要

| 检查项 | 状态 | 详情 |
|--------|------|------|
| 规范覆盖 | PASS/FAIL | X/Y 需求已覆盖 |
| 场景覆盖 | PASS/FAIL | X/Y 场景已实现 |
| 设计遵循 | PASS/FAIL | X/Y 决策已遵循 |
| 排除范围 | PASS/FAIL | 未违反排除范围 |

### 问题列表

#### CRITICAL（必须修复）

- [问题描述]
  - 规范位置：`openspec/specs/xxx/spec.md:Lxx`
  - 期望行为：...
  - 实际行为：...
  - 修复建议：...

#### WARNING（建议修复）

- [问题描述]
  - 相关文件：`src/xxx.py:Lxx`
  - 建议：...

#### SUGGESTION（可选改进）

- [改进建议]

### 最终结论

- [ ] 通过 — 实现与规范一致
- [ ] 需修复 — 存在 CRITICAL 问题
- [ ] 建议改进 — 存在 WARNING 但可接受
```

## 常见问题

| 问题 | 检查方法 |
|------|----------|
| "AI顺手多改了代码" | 检查排除范围，对比 git diff |
| "实现方式和 design.md 不一致" | 这是 bug，不是优化，需要修复 |
| "场景没有覆盖" | 检查测试文件是否存在对应测试 |
| "需求遗漏" | 搜索代码中是否有关键词实现 |

## 与代码审查的区别

| 维度 | 代码审查 | 规范合规审查 |
|------|----------|--------------|
| 关注点 | 代码质量 | 规范一致性 |
| 检查内容 | 命名、结构、错误处理 | 需求覆盖、场景实现、决策遵循 |
| 参考文档 | 代码规范、最佳实践 | OpenSpec 规范文档 |
| 失败含义 | 代码质量问题 | 方向偏离问题 |

两者互补，不可替代。

## 触发时机

- 完成代码实现后
- 代码审查之前
- 运行 `/opsx:verify` 之前
- 归档变更之前

## 护栏

- 必须读取规范文档，不能凭记忆判断
- 发现问题必须记录，不能忽略
- CRITICAL 问题必须修复后才能继续
- 不确定的问题标记为 WARNING 并说明原因
