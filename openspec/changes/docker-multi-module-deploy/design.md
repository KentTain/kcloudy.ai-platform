## 上下文

### 背景

前端模块系统已实现运行时模块注册和依赖解析。但当前构建和部署仍为单体方式：

1. 所有模块代码都打包到一个构建产物中
2. 无法选择模块组合
3. 所有模块部署在同一个域名下
4. 无法实现独立部署单个模块

需要实现构建时模块选择和多域名部署能力。

### 当前状态

- 单体 Vite 配置，打包所有模块
- Nginx 作为反向代理
- 单 Docker Compose 服务

### 约束

- 保持构建过程必须向后兼容：默认打包所有模块
- 支持 3 种部署模式：单体、分离、混合
- 保持开发体验不变：pnpm dev 仍加载所有模块
- 支持 Docker Compose 编排

### 利益相关者

- DevOps 团队：部署和运维
- 前端开发：构建配置
- 后端开发：API 跨域配置

## 目标 / 非目标

**目标：**

- 实现构建时模块选择，通过 BUILD_MODULES 环境变量控制
- 实现 Docker 多阶段构建优化，减小镜像大小
- 实现 Docker Compose 多服务配置，支持独立域名部署
- 实现模块间 API 代理和跨域处理
- 提供部署文档和示例

**非目标：**

- 模块热更新和运行时模块加载
- 微前端架构（Module Federation）
- 跨模块状态共享
- 运行时模块版本管理

## 决策

### 决策 1：构建时模块选择机制

**选择：Vite 插件 + 虚拟模块 + tree-shaking

实现方式：

1. Vite 插件读取 BUILD_MODULES 环境变量
2. 生成虚拟模块 src/config/modules.ts，导出选中的模块列表
3. main.ts 只注册列表中的模块
4. 未选中模块通过 tree-shaking 移除

**理由：**

- 利用 Vite 插件机制，侵入性小
- Tree-shaking 自动移除未使用模块代码
- TypeScript 类型安全
- 开发环境可保持所有模块可用

**备选方案：** 使用动态 import + 条件加载

- **放弃理由：** 运行时加载，tree-shaking 效果差

### 决策 2：Docker 多阶段构建

**选择：** 多阶段构建，按模块分割

**阶段划分：**

1. **Base：** 基础依赖安装
2. **Builder：** 代码编译，支持 BUILD_MODULES 参数
3. **Runtime：** Nginx 运行时，只包含构建产物

**理由：**

- 减小最终镜像大小（只保留运行时依赖）
- 利用 Docker 缓存层加速构建
- 支持不同 BUILD_MODULES 组合构建不同镜像

### 决策 3：部署模式选择

**选择：** 支持 3 种部署模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **单体部署** | 所有模块打包到一个镜像，单域名 | 开发、测试 |
| **分离部署** | 每个模块独立镜像，独立域名 | 生产、大团队 |
| **混合部署** | 部分模块合并，部分独立 | 中型团队 |

**理由：**

- 灵活适应不同部署需求
- 渐进式迁移：从单体到分离
- 支持不同团队的协作模式

### 决策 4：多域名路由策略

**选择：** Nginx 反向代理 + 路径前缀

**理由：**

- Nginx 反向代理成熟稳定
- 路径前缀路由简单可靠
- 支持 SSL 终止和负载均衡

**备选方案：** 子路径路由（example.com/demo）
- **放弃理由：** Vue Router history 模式配置复杂

### 决策 5：模块间 API 通信

**选择：** 统一 API 网关 + 内部服务调用

**理由：**

- 所有前端通过统一网关调用后端
- 避免跨域问题
- 统一认证和权限控制

### 决策 6：菜单聚合策略

**选择：** 主域名聚合所有模块菜单

**实现方式：**

1. 主域名（app.example.com）部署 framework 模块
2. framework 调用后端菜单 API 获取所有模块菜单
3. 点击菜单项跳转到对应模块域名

**理由：**

- 用户体验统一，一个入口访问所有功能
- 后端菜单支持权限控制
- 各模块独立部署

## 风险 / 权衡

### 风险 1：构建时模块选择导致运行时错误

**缓解措施：**

- 模块依赖检查：构建时验证依赖完整性
- 类型检查：TypeScript 编译检查
- 运行时降级：未找到模块时友好提示

### 风险 2：多域名跨域 Cookie 问题

**缓解措施：**

- 使用统一父域名 Cookie
- 配置 Nginx CORS 头
- 使用 Token 认证而非 Session

### 风险 3：多服务部署复杂度增加

**缓解措施：**

- 提供 Docker Compose 一键启动配置
- 提供部署文档和示例
- 保持单体部署作为默认选项
