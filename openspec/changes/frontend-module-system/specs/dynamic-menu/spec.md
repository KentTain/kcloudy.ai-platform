## 新增需求

### 需求:菜单 Store

系统必须提供菜单 Store（Pinia），管理动态菜单状态。

菜单 Store 必须包含以下状态：
- `menuItems`: 菜单树数组
- `loading`: 加载状态
- `error`: 错误信息

菜单 Store 必须包含以下方法：
- `fetchMenu()`: 从后端 API 获取菜单
- `getLocalMenu()`: 获取本地菜单（fallback）
- `refreshMenu()`: 刷新菜单

#### 场景:初始化菜单 Store
- **当** 应用启动时
- **那么** `menuItems` 为空数组，`loading` 为 false

#### 场景:从 API 加载菜单成功
- **当** 调用 `fetchMenu()` 且 API 返回成功
- **那么** `menuItems` 更新为返回的菜单树，`loading` 为 false

#### 场景:从 API 加载菜单失败
- **当** 调用 `fetchMenu()` 且 API 返回失败
- **那么** 自动 fallback 到本地菜单，`error` 记录错误信息

### 需求:菜单数据格式

系统必须支持统一的菜单数据格式：

``typescript
interface MenuItem {
  id: string;
  name: string;
  path: string;
  icon?: string;
  children?: MenuItem[];
  meta?: Record<string, unknown>;
}
``

菜单树必须支持多级嵌套，children 字段递归包含子菜单。

#### 场景:渲染一级菜单
- **当** 菜单树包含一级菜单项
- **那么** 侧边栏渲染一级菜单

#### 场景:渲染多级菜单
- **当** 菜单项包含 children
- **那么** 渲染为可展开的子菜单

### 需求:菜单权限过滤

系统必须根据用户权限过滤菜单项。

后端 API 必须返回用户有权限访问的菜单树。

前端禁止显示用户无权限的菜单项。

#### 场景:用户有部分权限
- **当** 用户只对 demo 和 iam 模块有权限
- **那么** 菜单树仅包含 demo 和 iam 相关菜单

#### 场景:用户无任何权限
- **当** 用户对任何菜单都无权限
- **那么** 菜单树为空数组

### 需求:菜单缓存

系统必须缓存菜单数据，避免重复请求。

菜单 Store 必须在首次加载后缓存数据，后续访问直接返回缓存。

用户切换租户或权限变更时，必须清除缓存并重新加载。

#### 场景:首次加载菜单
- **当** 首次调用 `fetchMenu()`
- **那么** 发起 API 请求

#### 场景:使用缓存菜单
- **当** 已加载过菜单，再次调用 `fetchMenu()`
- **那么** 直接返回缓存数据，不发起请求

#### 场景:清除缓存
- **当** 用户切换租户
- **那么** 清除菜单缓存，下次调用重新请求
