# 插件安装修改规范

本文档定义对现有插件安装功能的修改需求，将数据访问从 ORM 改为 Provider 模式。

## 修改需求

### 需求:插件包上传与解析

系统必须支持上传插件包（.zip 格式），并解析包内的 manifest.yaml 配置文件。

#### 场景:上传有效插件包
- **当** 管理员上传一个格式正确的插件包
- **那么** 系统解压并解析 manifest.yaml
- **且** 系统提取插件元数据（ID、名称、版本、作者等）
- **且** 系统验证插件配置的完整性

#### 场景:上传无效插件包
- **当** 管理员上传一个格式错误或损坏的插件包
- **那么** 系统拒绝安装并返回错误信息
- **且** 系统清理临时文件

#### 场景:上传缺少 manifest 的插件包
- **当** 管理员上传一个不包含 manifest.yaml 的插件包
- **那么** 系统拒绝安装并返回"缺少 manifest.yaml"错误

### 需求:插件安装

系统必须将插件安装到租户环境中，通过 Tenant Provider 创建安装记录，通过 AI 模块创建配置和运行时状态。

#### 场景:首次安装插件
- **当** 管理员安装一个新插件
- **那么** 系统通过 Tenant Provider 创建 PluginInstallation 记录（status=PENDING）
- **且** 系统在 AI 模块创建 plugin_configs 和 plugin_runtime_states 记录
- **且** 系统更新安装状态为 ACTIVE
- **且** 系统创建插件工作目录和虚拟环境
- **且** 系统上传插件资源到对象存储
- **且** 如果配置了 auto_start，系统启动插件

#### 场景:安装失败回滚
- **当** AI 侧创建配置失败
- **那么** 系统发布 PluginInstallationFailed 事件
- **且** Tenant 监听器删除安装记录
- **且** 系统清理临时文件和资源

#### 场景:安装已存在的插件版本
- **当** 管理员尝试安装相同 ID 和版本的插件
- **那么** 系统拒绝安装并返回"插件已安装"错误

#### 场景:安装不同版本的已存在插件
- **当** 管理员尝试安装相同 ID 但不同版本的插件
- **那么** 系统提示需要先卸载现有版本或使用升级功能

### 需求:插件升级

系统必须支持插件升级，保留原有配置并平滑过渡。

#### 场景:升级插件到新版本
- **当** 管理员上传新版本的插件包进行升级
- **那么** 系统验证新版本的 plugin_id 与现有插件一致
- **且** 系统卸载旧版本（保留凭证数据）
- **且** 系统通过 Provider 更新 Tenant 侧安装记录
- **且** 系统更新 AI 侧配置记录
- **且** 系统保留 auto_start 配置

#### 场景:升级失败时回滚
- **当** 插件升级过程中发生错误
- **那么** 系统尝试恢复旧版本的安装记录
- **且** 系统记录升级失败的详细信息

### 需求:插件卸载

系统必须支持插件卸载，通过 Provider 删除安装记录，清理 AI 侧配置和运行时状态。

#### 场景:卸载已安装的插件
- **当** 管理员卸载一个插件
- **那么** 系统停止正在运行的插件进程
- **且** 系统删除 AI 侧 plugin_configs 和 plugin_runtime_states 记录
- **且** 系统通过 Tenant Provider 删除安装记录
- **且** 系统删除插件工作目录和虚拟环境
- **且** 系统更新插件定义的 refers 计数（减 1）

#### 场景:卸载不存在的插件
- **当** 管理员尝试卸载一个不存在的插件
- **那么** 系统返回"插件不存在"错误

### 需求:插件列表查询

系统必须支持按条件查询插件列表，通过 Provider 获取安装记录，从 AI 侧获取配置信息。

#### 场景:查询租户的所有插件
- **当** 用户请求插件列表
- **那么** 系统通过 Provider 获取当前租户的所有安装记录
- **且** 系统从 AI 侧 plugin_configs 获取配置信息
- **且** 系统从 AI 侧 plugin_runtime_states 获取运行时状态
- **且** 每个插件包含 ID、名称、版本、状态等基本信息

#### 场景:按状态过滤插件
- **当** 用户请求特定状态的插件列表
- **那么** 系统返回匹配状态的插件列表

#### 场景:按类型过滤插件
- **当** 用户请求特定类型的插件列表（如 model、tool）
- **那么** 系统返回匹配类型的插件列表

### 需求:插件详情查询

系统必须支持查询插件的详细信息，包括配置、状态和运行时信息。

#### 场景:查询插件详情
- **当** 用户请求某个插件的详情
- **那么** 系统通过 Provider 获取安装记录
- **且** 系统从 AI 侧获取 plugin_config 和 runtime_state
- **且** 返回插件完整信息，包含插件配置、运行状态、调用统计等

### 需求:插件状态管理使用 Provider

系统必须通过 Tenant Provider 管理插件安装记录的状态字段。

#### 场景:更新安装状态为激活
- **当** AI 模块成功完成插件配置
- **那么** 系统通过 Provider 更新安装记录状态为 ACTIVE

#### 场景:更新安装状态为失败
- **当** AI 模块插件配置失败
- **那么** 系统通过 Provider 更新安装记录状态为 FAILED

#### 场景:禁用已激活的插件
- **当** 管理员通过 Web API 禁用插件
- **那么** 系统通过 Provider 更新安装记录状态从 ACTIVE 为 INACTIVE

#### 场景:启用已禁用的插件
- **当** 管理员通过 Web API 启用插件
- **那么** 系统通过 Provider 更新安装记录状态从 INACTIVE 为 ACTIVE

## 新增需求

### 需求:插件配置管理

系统必须提供插件配置的查询和更新接口。

#### 场景:查询插件配置
- **当** 用户请求插件配置
- **那么** 系统从 AI 侧 plugin_configs 表获取配置
- **且** 返回 plugin_config（能力配置）和 runtime_config（运行时配置）

#### 场景:更新运行时配置
- **当** 用户更新插件的 runtime_config
- **那么** 系统更新 AI 侧 plugin_configs 表
- **且** 返回更新时间

### 需求:插件运行时状态监控

系统必须提供插件运行时状态监控接口。

#### 场景:查询单个插件运行时状态
- **当** 用户请求插件运行时状态
- **那么** 系统从 AI 侧 plugin_runtime_states 表获取状态
- **且** 返回进程信息（PID、端口、工作目录）
- **且** 返回调用统计（调用次数、错误次数）
- **且** 返回时间戳（启动时间、最后访问时间、冻结时间）

#### 场景:批量查询运行时状态
- **当** 用户请求所有插件运行时状态
- **那么** 系统返回每个插件的状态摘要
- **且** 返回汇总统计（运行中进程数、总内存占用）
