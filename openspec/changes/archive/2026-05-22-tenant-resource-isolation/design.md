## Context

当前系统实现的是租户级逻辑隔离：
- **数据库**：共享数据库，通过 `tenant_id` 字段过滤
- **缓存**：共享 Redis，通过 Key 前缀隔离（`{tenant_id}:key`）
- **存储**：共享 MinIO Bucket，通过路径前缀隔离（`{tenant_id}/path`）

这种方案无法满足高安全要求租户的需求，需要升级为物理隔离方案。

### 约束条件

- **平滑迁移**：现有租户不能受影响，新租户可选择物理隔离
- **连接池管理**：数据库连接池有限，需要缓存和回收机制
- **敏感信息**：数据库密码需要加密存储

## Goals / Non-Goals

**Goals:**

1. 为租户提供独立的数据库、存储 Bucket、Redis DB 物理隔离能力
2. 实现连接池缓存管理，支持惰性加载和 LRU 回收
3. 支持平滑迁移，未配置独立资源的租户继续使用逻辑隔离
4. 敏感配置信息（数据库密码）加密存储

**Non-Goals:**

1. 不支持跨云厂商的资源配置（当前仅支持 PostgreSQL、MinIO、Redis）
2. 不实现完全独立的 Redis 实例隔离（仅支持 DB 编号隔离）
3. 不实现自动化迁移工具（租户从逻辑隔离迁移到物理隔离需要手动操作）

## Decisions

### 决策 1：数据库连接池缓存策略

**选择方案**：惰性加载 + LRU 回收

**备选方案**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 惰性加载 + LRU | 首次访问创建，超时回收 | 内存可控，按需创建 | 首次访问延迟 |
| 预加载 | 启动时加载所有租户 | 无冷启动 | 内存占用高，新租户需重启 |
| 无缓存 | 每次创建新连接 | 简单 | 性能差，连接风暴 |

**选择理由**：
- 租户数量可能较多，预加载会占用大量内存
- 大部分请求集中在少数活跃租户，LRU 可有效管理内存
- 首次访问延迟可接受（毫秒级）

### 决策 2：默认资源处理策略

**选择方案**：回退到逻辑隔离

**行为定义**：

| 资源 | 配置状态 | 行为 |
|------|----------|------|
| 数据库 | `db_name` 为空 | 使用默认数据库 + `tenant_id` 过滤 |
| 存储 | `storage_bucket` 为空 | 使用默认 Bucket + 路径前缀 |
| 缓存 | `cache_db` 为空 | 使用默认 Redis DB + Key 前缀 |

**选择理由**：
- 完全向后兼容，现有租户无需任何修改
- 新租户可按需启用物理隔离
- 降低运维复杂度

### 决策 3：敏感信息加密方案

**选择方案**：AES-256-GCM 加密 + 租户级密钥

**架构**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        加密架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Tenant 表：                                                   │
│   ├─ encryption_key: str   # 每个租户独立的加密密钥（主密钥加密）│
│   └─ db_password: str      # AES-256-GCM 加密存储              │
│                                                                 │
│   主密钥来源（优先级）：                                         │
│   1. 环境变量 TENANT_ENCRYPTION_MASTER_KEY                      │
│   2. 配置文件 security.encryption.master_key                    │
│   3. 应用启动时自动生成（仅适用于开发环境）                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**备选方案**：

| 方案 | 描述 | 安全性 | 复杂度 |
|------|------|--------|--------|
| AES-256-GCM | 应用层加密 | 高 | 中 |
| Vault 集成 | 外部密钥管理 | 最高 | 高 |
| 明文存储 | 无加密 | 低 | 低 |

**选择理由**：
- 安全性足够满足大多数场景
- 实现复杂度适中
- 后续可扩展 Vault 集成

## Risks / Trade-offs

### 风险 1：连接池资源耗尽

**风险描述**：大量租户同时活跃可能导致连接池资源耗尽

**缓解措施**：
- 设置最大缓存连接池数量限制（如 50 个）
- 实现 LRU 回收，优先释放不活跃的连接池
- 监控连接池使用情况，告警阈值触发

### 风险 2：Redis DB 数量限制

**风险描述**：Redis 默认仅支持 16 个 DB，无法支持大量租户

**缓解措施**：
- 物理隔离为可选功能，仅高安全需求租户启用
- 超过限制时回退到逻辑隔离（Key 前缀）
- 未来可扩展独立 Redis 实例支持

### 风险 3：密钥丢失导致数据不可访问

**风险描述**：租户加密密钥丢失后，数据库密码无法解密

**缓解措施**：
- 主密钥备份到安全存储
- 提供 key rotation 机制
- 文档说明密钥管理最佳实践

### 权衡：性能 vs 隔离性

- **物理隔离**：更高安全性，但连接池管理复杂，首次访问有延迟
- **逻辑隔离**：更简单高效，但安全性较低

**决策**：提供两种模式，让租户按需选择

## Migration Plan

### Phase 1：数据库迁移（无破坏性）

1. 创建迁移脚本添加新字段（所有字段允许 NULL）
2. 现有租户数据无需修改
3. 新字段通过应用层填充

### Phase 2：Framework 模块扩展

1. 创建 `DatabaseEnginePool`
2. 创建 `TenantStorageManager`
3. 创建 `TenantCacheManager`
4. 更新 `TenantContext`

### Phase 3：IAM 模块扩展

1. 扩展 `Tenant` 模型
2. 更新 `TenantService`
3. 更新 `TenantProvider` 实现

### Phase 4：测试与验证

1. 单元测试
2. 集成测试
3. 性能测试

## Open Questions
