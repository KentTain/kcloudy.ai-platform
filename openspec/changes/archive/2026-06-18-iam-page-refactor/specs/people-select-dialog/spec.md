## 新增需求

### 需求:人员选择器弹窗组件
系统必须提供一个通用的 PeopleSelectDialog 组件，放置于 `src/components/common/feedback/` 目录，支持通过组织树浏览并选择人员。

#### 场景:单选模式打开弹窗
- **当** 调用方传入 `multiple: false`
- **那么** 弹窗打开并显示组织树，用户只能选择一个人员，选择后自动确认

#### 场景:多选模式打开弹窗
- **当** 调用方传入 `multiple: true`
- **那么** 弹窗打开并显示组织树，用户可以选择多个人员，点击"确定"按钮确认

#### 场景:禁用已选人员
- **当** 调用方传入 `disabledIds` 列表
- **那么** 列表中的人员在树中显示为禁用状态，不可勾选

#### 场景:确认选择
- **当** 用户在弹窗中完成选择并点击"确定"
- **那么** 弹窗触发 `confirm` 事件，携带选中的人员 ID 列表和人员对象列表，并关闭

### 需求:组织人员树 composable
系统必须提供 `usePeopleTree` composable，封装组织+人员树的无头逻辑。

#### 场景:加载根节点
- **当** composable 初始化
- **那么** 自动调用 API 加载根级组织节点并展开第一层

#### 场景:懒加载子节点
- **当** 用户展开一个组织节点
- **那么** 如果该节点有子组织或有直属人员但尚未加载，调用 API 获取子节点数据

#### 场景:三态复选框选中组织
- **当** 用户勾选一个组织节点的复选框
- **那么** 该组织下所有已加载的直属人员被选中；如果所有人员已选中则取消全选

#### 场景:搜索过滤
- **当** 用户输入搜索关键字
- **那么** 树节点只显示匹配的组织及其祖先路径

### 需求:组织人员树数据类型
系统必须定义 `OrgTreeNode`、`PeopleItem` 等前端类型，与后端 API 响应对齐。

#### 场景:类型对齐
- **当** 后端返回部门树接口数据
- **那么** 前端 `OrgTreeNode` 类型必须包含 `id`、`name`、`has_org_num`、`has_user_num`、`parent_id`、`tree_leaf`、`children` 字段
