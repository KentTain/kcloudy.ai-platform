## ADDED Requirements

### 需求:模块定义自动同步

系统必须在应用启动时自动将模块声明同步到数据库。

#### 场景:同步模块基本信息
- **当** 应用启动，模块定义了 get_module_definition() 返回 ModuleDefinition
- **那么** 系统创建或更新 tenant.modules 表中对应记录

#### 场景:同步模块菜单
- **当** 模块定义包含 menus 列表
- **那么** 系统创建或更新 tenant.module_menus 表中对应记录

#### 场景:同步菜单父子关系
- **当** 菜单定义了 parent_code 字段
- **那么** 系统解析父菜单 ID，正确设置 parent_id 字段

#### 场景:同步模块权限
- **当** 模块定义包含 permissions 列表
- **那么** 系统创建或更新 tenant.module_permissions 表中对应记录

#### 场景:同步默认角色
- **当** 模块定义包含 default_roles 列表
- **那么** 系统创建或更新 tenant.module_roles 表中对应记录，并创建角色权限关联

#### 场景:清理孤立菜单
- **当** 模块定义中删除了某个菜单
- **那么** 同步后数据库中该菜单被删除

#### 场景:清理孤立权限
- **当** 模块定义中删除了某个权限
- **那么** 同步后数据库中该权限及关联被删除

#### 场景:清理孤立角色
- **当** 模块定义中删除了某个默认角色
- **那么** 同步后数据库中该角色及关联被删除

### 需求:同步幂等性

系统必须保证同步操作的幂等性。

#### 场景:重复启动应用
- **当** 应用多次重启，模块定义未变化
- **那么** 数据库数据保持一致，不会产生重复记录

#### 场景:模块定义更新
- **当** 模块定义更新后重启应用
- **那么** 数据库数据更新为新定义

### 需求:同步失败处理

系统必须正确处理同步失败的情况。

#### 场景:菜单循环引用
- **当** 菜单 A 的 parent_code 指向 B，B 的 parent_code 指向 A
- **那么** 同步失败，应用启动报错，日志记录"菜单循环引用"

#### 场景:父菜单不存在
- **当** 菜单的 parent_code 指向不存在的菜单
- **那么** 同步失败，应用启动报错，日志记录"父菜单不存在"

#### 场景:角色权限 code 不存在
- **当** RoleDef 中的 permission_codes 引用不存在的权限
- **那么** 同步失败，应用启动报错，日志记录"权限 code 不存在"

### 需求:模块未实现声明

系统必须正确处理未实现 get_module_definition() 的模块。

#### 场景:模块未实现声明方法
- **当** 某模块未实现 get_module_definition() 或返回 None
- **那么** 同步服务跳过该模块，该模块的元数据保持手动管理
