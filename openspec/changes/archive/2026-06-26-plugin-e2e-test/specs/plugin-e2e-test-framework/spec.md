## 新增需求

### 需求:E2E 测试环境隔离

系统必须为每个 E2E 测试用例提供独立的测试环境，包括独立的测试租户、数据库 Schema、Redis 命名空间和 OSS Bucket。

#### 场景:创建测试租户

- **当** E2E 测试启动
- **那么** 系统创建唯一的测试租户（ID 格式为 `e2e-test-{timestamp}`）
- **且** 测试租户与其他租户数据完全隔离

#### 场景:清理测试资源

- **当** E2E 测试结束
- **那么** 系统删除测试租户的所有数据
- **且** 系统清理 Redis 中测试相关的 Key
- **且** 系统删除 OSS 中测试上传的文件

### 需求:E2E 测试夹具管理

系统必须提供可复用的测试夹具，简化测试用例编写。

#### 场景:获取测试插件包路径

- **当** 测试需要使用真实插件包
- **那么** 系统返回 `server/plugins/` 目录下插件包的绝对路径
- **且** 支持按插件 ID 获取指定插件包路径

#### 场景:获取测试 API Key

- **当** 测试需要调用真实 AI 模型
- **那么** 系统从环境变量读取 API Key
- **且** 如果未配置则跳过测试（pytest.skip）

#### 场景:创建测试数据库会话

- **当** 测试需要访问数据库
- **那么** 系统提供独立的数据库会话
- **且** 会话自动提交和关闭

### 需求:E2E 测试标记和配置

系统必须提供 pytest 标记用于识别和控制 E2E 测试。

#### 场景:标记 E2E 测试

- **当** 编写 E2E 测试用例
- **那么** 使用 `@pytest.mark.e2e` 标记测试
- **且** 默认测试运行时跳过 E2E 测试

#### 场景:运行 E2E 测试

- **当** 执行 `pytest -m e2e` 命令
- **那么** 系统仅运行标记为 e2e 的测试用例

### 需求:测试辅助工具类

系统必须提供 PluginTestHelper 工具类，封装常用测试操作。

#### 场景:等待插件状态

- **当** 测试需要等待插件达到指定状态
- **那么** 工具类提供 `wait_for_plugin_status` 方法
- **且** 支持超时配置和轮询间隔配置

#### 场景:断言插件安装成功

- **当** 测试需要验证插件安装结果
- **那么** 工具类提供 `assert_plugin_installed` 方法
- **且** 自动检查安装状态和数据库记录

#### 场景:断言插件运行正常

- **当** 测试需要验证插件运行状态
- **那么** 工具类提供 `assert_plugin_running` 方法
- **且** 检查进程存在且状态为 ACTIVE
