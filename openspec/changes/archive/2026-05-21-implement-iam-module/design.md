## Context

当前系统已有 `tenant` 模块，包含 `Tenant`、`TenantConfig`、`TenantAdmin`、`UserTenant` 四个数据模型。但这些模型仅定义了租户的基础结构，缺乏完整的用户认证、授权、组织架构等功能。

本次设计将：
1. 创建独立的 `iam` 模块，整合租户管理与用户身份管理
2. 迁移现有 tenant 数据模型到 iam 模块
3. 新增用户、角色、权限、部门等数据模型
4. 实现认证授权相关的 API 和服务层

## Goals / Non-Goals

**Goals:**

- 实现完整的用户认证流程（本地登录、OAuth2 登录、登出）
- 实现 JWT 双 Token 机制 + Redis 混合模式
- 实现 RBAC 权限控制（租户管理员、系统管理员、普通用户）
- 实现组织架构管理（部门）
- 支持微信、企微 OAuth2 登录
- 支持 OAuth 用户首次登录补全信息

**Non-Goals:**

- 多因素认证（MFA）— 后续迭代
- 多设备管理、踢人下线 — 后续迭代
- 操作审计日志 — 后续迭代
- 钉钉、QQ 等其他 OAuth2 提供商 — 后续迭代

## Decisions

### D1: 模块命名与结构

**决策**：使用 `iam` 作为模块名，含义为 Identity and Access Management。

**理由**：
- 业界标准术语（AWS IAM、阿里云 RAM）
- 涵盖身份管理（用户、租户、组织）和访问管理（认证、授权）
- 简短、语义清晰

**目录结构**：
```
demo/models/iam/
├── __init__.py
├── tenant.py          # 从 tenant 模块迁移
├── tenant_config.py   # 从 tenant 模块迁移
├── tenant_admin.py    # 从 tenant 模块迁移
├── user_tenant.py     # 从 tenant 模块迁移
├── user.py            # 新增
├── oauth_connection.py # 新增
├── department.py      # 新增
├── role.py            # 新增
├── permission.py      # 新增
└── enums.py           # 枚举定义
```

### D2: 密码存储策略

**决策**：前端明文传输（依赖 HTTPS）+ 后端 BCrypt 哈希存储。

**理由**：
- HTTPS 已提供传输层加密，前端 MD5 是多余的
- BCrypt 是业界标准，自带盐值，抗彩虹表攻击
- cost factor = 12，在安全性和性能之间取得平衡

**备选方案**：
- Argon2：更安全，但库支持较少，配置复杂
- 前端 MD5：安全性有限，MD5 值本身成为"密码"

### D3: JWT + Redis 混合模式

**决策**：Token 自包含基本权限信息 + Redis 存储会话版本号。

**理由**：
- 大多数请求无需查 Redis，性能好
- 权限变更时可实时生效（通过版本号机制）
- 支持登出时立即失效（Token 黑名单）

**验证流程**：
```
1. 解析 Token，获取 session_id 和 version
2. 查询 Redis 黑名单 → 存在则拒绝
3. version 匹配 → 直接使用 Token 权限
4. version 不匹配 → 查 Redis 更新权限，签发新 Token
```

### D4: OAuth 账号关联策略

**决策**：用户表 + OAuth 关联表（一对多）。

**理由**：
- 支持一个用户绑定多个第三方账号
- 扩展性好，新增 OAuth 提供商无需改用户表
- 清晰的数据边界，便于维护

### D5: RBAC 权限模型

**决策**：用户-角色-权限 三层模型。

**理由**：
- 灵活：用户可拥有多个角色，角色可包含多个权限
- 租户隔离：角色绑定租户，权限检查时考虑租户上下文
- 预定义角色：租户管理员、系统管理员、普通用户

**权限命名规范**：
- 格式：`资源:操作`（如 `user:read`、`user:write`）
- 支持通配符：`user:*` 表示用户的所有权限

### D6: API 前缀设计

**决策**：使用 `/api/v1/iam/` 作为业务用户 API 前缀。

**理由**：
- 与现有 `/admin/v1/` 前缀区分（管理后台）
- 统一的模块前缀便于路由和权限管理

## Risks / Trade-offs

### R1: Token 安全性

**风险**：Access Token 被窃取后，攻击者可在有效期内访问系统。

**缓解措施**：
- Access Token 有效期较短（2 小时）
- 敏感操作（修改密码、删除用户）需要二次验证
- 支持 Token 黑名单，用户可主动登出

### R2: OAuth 用户密码缺失

**风险**：OAuth 登录创建的用户没有密码，存在安全隐患。

**缓解措施**：
- OAuth 用户首次登录标记 `profile_completed = FALSE`
- 引导用户补全密码、邮箱等信息
- 未补全信息的用户限制部分敏感操作

### R3: 租户数据隔离

**风险**：多租户环境下，权限检查遗漏导致数据泄露。

**缓解措施**：
- 使用租户中间件自动注入租户上下文
- 数据模型层增加 `tenant_id` 字段
- 查询时自动添加租户过滤条件

### R4: 迁移兼容性

**风险**：将 tenant 模型迁移到 iam 模块可能影响现有代码。

**缓解措施**：
- 迁移后保留原导入路径的兼容性（重导出）
- 数据库表名不变，仅迁移代码位置
- 提供迁移脚本和测试用例

## Open Questions

1. **Refresh Token 存储方式**：存储在 Redis 还是仅使用 JWT 签名验证？
   - 倾向：存储在 Redis，便于撤销和跟踪

2. **权限缓存策略**：权限信息缓存多久？
   - 倾向：Redis 缓存 5 分钟，权限变更时清除缓存

3. **租户管理员初始化**：系统启动时如何创建默认租户管理员？
   - 倾向：通过数据初始化脚本（seed），账号密码从环境变量读取
