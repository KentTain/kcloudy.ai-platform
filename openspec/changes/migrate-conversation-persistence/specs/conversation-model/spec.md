# 会话模型规范

## 新增需求

### 需求:系统支持会话持久化

系统必须提供 `Conversation` ORM 模型，支持会话数据的持久化存储。

#### 场景:创建会话

- **当** 创建新的 `Conversation` 记录
- **那么** 系统必须生成 UUID 作为主键，并记录 `tenant_id`、`app_id`、`name`、`status` 等字段

#### 场景:会话归属于租户

- **当** 创建会话时
- **那么** 系统必须将会话关联到当前租户（`tenant_id`）

#### 场景:会话归属于应用

- **当** 创建会话时
- **那么** 系统必须将会话关联到指定应用（`app_id`）

---

### 需求:系统支持会话状态管理

系统必须支持会话状态枚举和状态变更。

#### 场景:会话状态枚举

- **当** 查询会话状态
- **那么** 系统必须支持 `normal`、`archived`、`deleted` 三种状态

#### 场景:归档会话

- **当** 调用会话归档操作
- **那么** 系统必须将会话状态从 `normal` 更新为 `archived`

#### 场景:删除会话

- **当** 调用会话删除操作
- **那么** 系统必须将会话状态更新为 `deleted`（软删除）

---

### 需求:系统支持会话查询

系统必须支持按条件查询会话列表。

#### 场景:按租户查询会话

- **当** 调用会话查询接口
- **那么** 系统必须只返回当前租户的会话记录

#### 场景:按应用查询会话

- **当** 指定 `app_id` 查询
- **那么** 系统必须只返回该应用的会话记录

#### 场景:按状态过滤会话

- **当** 指定 `status=normal` 过滤
- **那么** 系统必须只返回状态为 `normal` 的会话

---

### 需求:系统支持会话 CRUD 操作

系统必须提供会话的增删改查操作。

#### 场景:创建会话成功

- **当** 使用有效数据创建会话
- **那么** 系统必须返回创建的会话对象，包含生成的 `id` 和 `created_at` 时间戳

#### 场景:更新会话成功

- **当** 更新会话的 `name` 或 `status` 字段
- **那么** 系统必须更新 `updated_at` 时间戳

#### 场景:查询会话不存在

- **当** 查询不存在的会话 ID
- **那么** 系统必须返回 `None` 或抛出 `NotFoundError`

---

### 需求:系统支持租户隔离

系统必须确保不同租户的会话数据相互隔离。

#### 场景:禁止跨租户访问

- **当** 尝试访问其他租户的会话
- **那么** 系统必须拒绝访问并返回空结果或错误

#### 场景:查询自动过滤租户

- **当** 在租户上下文中查询会话列表
- **那么** 系统必须自动添加 `tenant_id` 过滤条件
