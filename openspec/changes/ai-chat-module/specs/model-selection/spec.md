## 新增需求

### 需求:用户可以选择不同的模型提供商

用户必须能够从支持的模型提供商列表中选择一个提供商。

#### 场景:显示提供商列表
- **当** 用户打开模型选择器
- **那么** 系统显示可用的模型提供商列表

#### 场景:选择提供商
- **当** 用户选择一个提供商
- **那么** 系统更新当前模型配置
- **那么** 后续对话使用新选择的提供商

### 需求:用户可以选择不同的模型

用户必须能够在选定提供商下选择具体的模型。

#### 场景:显示模型列表
- **当** 用户选择了提供商
- **那么** 系统显示该提供商支持的模型列表

#### 场景:选择模型
- **当** 用户选择一个模型
- **那么** 系统更新当前模型配置
- **那么** 后续对话使用新选择的模型

### 需求:模型配置必须传递到后端

前端必须将用户选择的模型配置传递到后端。

#### 场景:模型配置传递
- **当** 用户发送消息
- **那么** 请求体的 `body.model` 包含当前选择的模型配置
- **那么** 配置包含 `provider`、`name`、`completionParams`

#### 场景:后端使用指定模型
- **当** 后端收到带有模型配置的请求
- **那么** 系统使用指定的提供商和模型创建 `AlonChatModel`
- **那么** 系统应用指定的模型参数

### 需求:模型选择状态必须持久化

用户的模型选择必须在会话期间保持，并在下次访问时恢复。

#### 场景:模型选择持久化
- **当** 用户选择模型
- **那么** 系统保存选择到本地存储或会话状态
- **那么** 刷新页面后选择仍然保持
