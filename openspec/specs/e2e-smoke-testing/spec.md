## 新增需求

### 需求:菜单驱动的页面可访问性验证

系统必须能够从 `/me` 接口获取菜单数据，自动遍历可见菜单项，验证每个页面的可访问性。

#### 场景:遍历所有可见菜单
- **当** 执行冒烟测试
- **那么** 系统从 `/api/tenant/admin/v1/admin/me` 获取 `menus` 数组
- **那么** 系统过滤 `is_visible === true` 的菜单项
- **那么** 系统按 `sort_order` 排序后依次访问每个菜单的 `path`
- **那么** 每个页面必须返回 HTTP 200 且无 JavaScript 错误

#### 场景:验证页面基础渲染
- **当** 访问菜单对应的页面路径
- **那么** 页面骨架屏必须在 10 秒内消失
- **那么** 页面必须有可见的内容区域
- **那么** 页面不得有控制台错误（允许 warning）

### 需求:统计卡片渲染验证

系统必须验证页面中的统计卡片是否正确渲染。

#### 场景:验证统计卡片存在
- **当** 页面包含统计卡片区域
- **那么** 卡片标题必须可见（如"租户总数"、"未激活数"）
- **那么** 卡片数值必须是有效的数字格式

### 需求:Tab 页签切换验证

系统必须验证多 Tab 页面的切换功能。

#### 场景:资源配置页面 Tab 切换
- **当** 访问 `/admin/resources` 页面
- **那么** 页面必须显示 5 个 Tab（数据库、存储、缓存、队列、发布订阅）
- **那么** 默认选中第一个 Tab
- **那么** 点击其他 Tab 后列表数据正确加载

### 需求:测试场景映射配置

系统必须支持将菜单路径映射到具体的测试场景。

#### 场景:配置测试场景覆盖
- **当** 定义 `MENU_TESTS` 配置对象
- **那么** 每个菜单路径可关联多个测试场景名称
- **那么** 测试框架自动为每个场景生成独立的测试用例
