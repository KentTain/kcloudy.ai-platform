## 新增需求

### 需求:自动检测测试覆盖缺失

系统必须能够自动检测菜单与测试用例的覆盖差异，生成缺失报告。

#### 场景:获取当前菜单列表
- **当** 运行缺失检测脚本
- **那么** 脚本向 `/api/tenant/admin/v1/auth/login` 发送请求获取 Token
- **那么** 脚本向 `/api/tenant/admin/v1/admin/me` 获取菜单数据
- **那么** 提取所有 `is_visible === true` 的菜单项

#### 场景:扫描现有测试用例
- **当** 脚本扫描 `tests/tenant/e2e/*.spec.ts` 文件
- **那么** 使用正则表达式提取所有 `test('xxx', ...)` 的测试名称
- **那么** 建立测试名称与文件的映射关系

#### 场景:生成缺失报告
- **当** 完成菜单与测试的对比分析
- **那么** 在 `docs/tests/` 目录创建 `test-lose-item-{YYYY-MM-DD}.md` 文件
- **那么** 报告包含：菜单名称、路径、测试文件、覆盖状态
- **那么** 报告包含：缺失测试清单（逐项列出）

### 需求:报告格式规范

系统必须生成标准化的 Markdown 格式报告。

#### 场景:报告结构完整
- **当** 生成缺失报告
- **那么** 报告必须包含标题、生成时间、当前菜单列表表格
- **那么** 报告必须包含缺失测试清单（按菜单分组）
- **那么** 表格使用 ✅/⚠️/❌ 标记覆盖状态

#### 场景:增量报告
- **当** 同一天多次运行检测脚本
- **那么** 覆盖已有报告文件（不创建多个版本）

### 需求:脚本独立运行

系统必须支持脚本独立运行，不依赖 Playwright 测试环境。

#### 场景:使用 Node.js 直接执行
- **当** 执行 `node scripts/e2e-check-coverage.ts`
- **那么** 脚本使用 axios 发送 HTTP 请求
- **那么** 脚本使用 glob 扫描测试文件
- **那么** 脚本使用 fs 写入报告文件

#### 场景:环境变量配置
- **当** 设置 `E2E_API_BASE` 环境变量
- **那么** 脚本使用该变量作为 API 基础地址
- **那么** 默认使用 `http://localhost:8080/api`
