## 新增需求

### 需求:插件进程隔离运行

系统必须为每个插件创建独立的进程运行环境，确保插件之间的资源隔离和故障隔离。

#### 场景:启动插件进程
- **当** 管理员启动一个已安装的插件
- **那么** 系统创建独立的子进程运行插件代码
- **且** 插件进程拥有独立的工作目录和虚拟环境

#### 场景:插件进程崩溃不影响主服务
- **当** 插件进程因异常崩溃
- **那么** 主服务进程继续运行
- **且** 系统记录崩溃日志和错误信息
- **且** 系统更新插件状态为 INACTIVE

### 需求:插件虚拟环境管理

系统必须使用 uv 为每个插件创建独立的 Python 虚拟环境，确保插件依赖与主应用隔离。

#### 场景:安装插件时创建虚拟环境
- **当** 安装一个包含 Python 依赖的插件
- **那么** 系统使用 uv 创建虚拟环境
- **且** 系统安装插件声明的依赖包
- **且** 虚拟环境路径记录在插件安装记录中

#### 场景:卸载插件时清理虚拟环境
- **当** 卸载一个插件
- **那么** 系统删除该插件的虚拟环境目录
- **且** 系统释放磁盘空间

### 需求:插件通信协议

系统必须使用 JSON 协议与插件进程通信，支持请求-响应和流式两种模式。

#### 场景:同步调用插件方法
- **当** 系统调用插件的某个方法
- **那么** 系统通过 stdin 发送 JSON 格式的请求
- **且** 插件通过 stdout 返回 JSON 格式的响应
- **且** 请求包含唯一的 session_id 用于匹配响应

#### 场景:流式调用插件方法
- **当** 系统以流式模式调用插件
- **那么** 插件可以返回多个增量响应
- **且** 每个响应包含相同的 session_id
- **且** 最后一个响应标记为完成

### 需求:插件生命周期管理

系统必须支持插件的启动、停止、重启操作，并正确管理进程状态。

#### 场景:启动已安装的插件
- **当** 管理员请求启动插件
- **那么** 系统检查插件是否已在运行
- **且** 如果未运行，系统创建并启动插件进程
- **且** 系统更新插件的 last_started_at 时间戳

#### 场景:停止运行中的插件
- **当** 管理员请求停止插件
- **那么** 系统发送终止信号给插件进程
- **且** 系统等待进程正常退出（最多 30 秒）
- **且** 如果超时，系统强制终止进程
- **且** 系统更新插件的 last_stopped_at 时间戳

### 需求:插件懒加载与预热

系统必须支持插件懒加载模式，并提供预热机制减少首次调用延迟。

#### 场景:懒加载模式首次调用
- **当** 首次调用一个未启动的插件
- **那么** 系统自动启动插件进程
- **且** 系统等待插件就绪后执行调用

#### 场景:启动时预热指定插件
- **当** 系统启动时配置了插件预热
- **那么** 系统在后台启动预热列表中的插件
- **且** 预热结果记录在日志中

#### 场景:自动冻结长期未使用的插件
- **当** 插件超过冻结阈值时间未被访问
- **那么** 系统自动停止该插件以释放资源
- **且** 系统更新插件状态为 INACTIVE

### 需求:AI 模块管理插件配置

AI 模块必须在 AI Schema 中维护插件配置表（`plugin_configs`），存储插件能力定义和持久化运行时配置。

#### 场景:创建插件配置

- **当** 插件安装成功
- **那么** AI 模块必须在 `plugin_configs` 表中创建记录，包含 `plugin_config` 和 `runtime_config` 字段

#### 场景:更新插件配置

- **当** 插件配置发生变化
- **那么** AI 模块必须更新 `plugin_configs` 表中对应的 `plugin_config` 或 `runtime_config` 字段

#### 场景:删除插件配置

- **当** 插件卸载
- **那么** AI 模块必须删除 `plugin_configs` 表中对应的记录

### 需求:AI 模块管理运行时状态

AI 模块必须在 AI Schema 中维护运行时状态表（`plugin_runtime_states`），存储进程信息、调用统计和冻结状态。

#### 场景:创建运行时状态

- **当** 插件安装成功
- **那么** AI 模块必须在 `plugin_runtime_states` 表中创建初始记录

#### 场景:更新进程信息

- **当** 插件启动或停止
- **那么** AI 模块必须更新 `process_id`、`port`、`last_started_at` 或 `last_stopped_at` 字段

#### 场景:更新调用统计

- **当** 插件被调用
- **那么** AI 模块必须递增 `call_count` 字段，更新 `last_accessed_at` 时间戳

#### 场景:记录错误信息

- **当** 插件调用失败
- **那么** AI 模块必须递增 `error_count` 字段，记录 `last_error` 信息

#### 场景:更新冻结状态

- **当** 插件被冻结或解冻
- **那么** AI 模块必须更新 `frozen_at` 字段

### 需求:通过 Provider 访问安装记录

AI 模块必须通过 `PluginInstallationProvider` 协议访问 Tenant 侧的安装记录，不得直接访问 Tenant Schema 的数据表。

#### 场景:加载插件元数据

- **当** PluginManager 启动时加载插件
- **那么** AI 模块必须调用 `Provider.get_tenant_installations()` 获取安装记录列表

#### 场景:保存安装记录

- **当** PluginManager 安装插件
- **那么** AI 模块必须调用 `Provider.create_installation()` 创建 Tenant 侧记录

#### 场景:更新安装状态

- **当** 安装流程完成或失败
- **那么** AI 模块必须调用 `Provider.update_installation()` 更新状态

### 需求:事件驱动一致性

AI 模块必须在安装或卸载失败时发布事件，触发 Tenant 侧的补偿操作。

#### 场景:安装失败发布事件

- **当** AI 侧配置创建失败，但 Tenant 侧安装记录已存在
- **那么** AI 模块必须发布 `PluginInstallationFailed` 事件，包含租户 ID、插件 ID 和错误信息

#### 场景:卸载失败发布事件

- **当** AI 侧数据删除失败，但 Tenant 侧记录已删除
- **那么** AI 模块必须发布 `PluginUninstallFailed` 事件

### 需求:PluginManager 数据访问方式

PluginManager 必须通过 `PluginInstallationProvider` 获取 Tenant 侧安装记录，并查询 AI 侧 `plugin_configs` 和 `plugin_runtime_states` 表组装完整的插件信息。

#### 场景:加载插件元数据（使用 Provider）

- **当** PluginManager 执行 `_load_plugins_metadata_from_database()`
- **那么** 系统必须：
  1. 调用 `Provider.get_tenant_installations()` 获取安装记录
  2. 查询 `ai.plugin_configs` 获取配置
  3. 查询 `ai.plugin_runtime_states` 获取运行时状态
  4. 组装 `PluginInfo` 对象

#### 场景:保存安装记录（使用 Provider）

- **当** PluginManager 执行 `_save_plugin_installation_to_database()`
- **那么** 系统必须：
  1. 调用 `Provider.create_installation()` 创建 Tenant 侧记录
  2. 在当前事务中创建 `ai.plugin_configs` 和 `ai.plugin_runtime_states` 记录
