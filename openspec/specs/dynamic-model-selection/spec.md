## 新增需求

### 需求:用户可以动态选择模型

系统必须提供动态模型选择器，展示当前可用的模型列表。

#### 场景:模型列表加载
- **当** 用户打开模型选择器
- **那么** 系统请求 `GET /api/v1/models`
- **那么** 系统展示按提供商分组的模型列表
- **那么** 每个模型展示名称和描述

#### 场景:模型选择生效
- **当** 用户选择某个模型
- **那么** 系统更新当前对话使用的模型
- **那么** 后续消息使用选定的模型生成响应

#### 场景:模型选择持久化
- **当** 用户选择模型后刷新页面
- **那么** 系统保持用户选择的模型设置
- **那么** 模型选择器正确回显当前选择

### 需求:后端必须返回动态模型列表

后端必须提供接口返回已配置的提供商和模型列表。

#### 场景:返回模型列表
- **当** 前端请求 `GET /api/v1/models`
- **那么** 后端返回已配置的提供商列表
- **那么** 每个提供商包含其可用模型列表
- **那么** 响应格式为 `{ providers: [{ id, name, models: [...] }] }`

#### 场景:仅返回已启用的模型
- **当** 后端遍历提供商配置
- **那么** 仅返回已启用且配置完整的模型
- **那么** 不暴露敏感凭证信息

### 需求:模型选择必须租户隔离

模型列表必须根据租户配置返回可用模型。

#### 场景:租户模型配置
- **当** 租户未配置某提供商
- **那么** 模型列表不展示该提供商
- **那么** 用户无法选择未授权的模型
